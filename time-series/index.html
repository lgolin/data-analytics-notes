1 What is Time Series Analysis? The Big Picture

1.1 The core idea in one sentence

Time series analysis is the art of listening to data over time, understanding its patterns, and using those patterns to predict the future.

1.2 A real-world analogy to make it stick

Imagine you work at a coffee shop and every day you record how many coffees you sell. After a year, you notice:

  • Sales have been growing overall (more customers each month) → this is Trend
  • Sales spike every Monday morning and dip on Sundays → this is Seasonality
  • One week sales dropped because of a snowstorm → this is Random noise

Time series analysis is exactly this: you look at your coffee sales data, pull apart these different layers, understand what drives each one, and then use that understanding to say “next month we’ll probably sell around X coffees.”

1.3 Why does it matter beyond this module?

This thinking transfers to any subject or job that involves data collected over time:

Field Time Series application
Finance Predicting stock prices, exchange rates
Healthcare Tracking disease spread over weeks
Marketing Forecasting monthly sales
Engineering Monitoring machine performance
Economics Modelling GDP, inflation, unemployment
Climate Science Analysing temperature changes

The concepts you learn here — patterns, prediction, uncertainty — are universal.

2 The Four Components of Any Time Series

2.1 The recipe analogy

Think of a time series like a recipe. A cake is made of flour, sugar, butter, and eggs — but you only see the finished cake. Time series analysis is like reverse-engineering a recipe: you have the finished cake (your data), and you want to figure out how much of each ingredient went in.

The “ingredients” of any time series are:

2.1.1 Trend (T) — The long-term direction

What it is: The overall movement of your data upwards or downwards over a long period.

Example: Global average temperatures have been rising for 100 years. If you plot this, the long-term direction is upward — that upward slope is the trend.

Key insight: A trend doesn’t have to be a straight line. It can curve, slow down, or reverse.

2.1.2 Seasonality (S) — The repeating pattern

What it is: A regular, predictable pattern that repeats within a fixed period (usually within one year).

Example: Ice cream sales are always high in summer and low in winter — every single year, without fail. This is seasonal behaviour.

Key insight: Seasonality is predictable and regular. It completes itself within a calendar year.

2.1.3 Cyclical Component (C) — The long, irregular wave

What it is: Patterns that repeat over multiple years, but are not as regular as seasons. Usually linked to economic cycles.

Example: After the 2009 financial crisis, nearly every economic indicator dropped. This is a cyclical downturn — it lasts years, not months, and cannot be predicted with the same precision as seasons.

Key difference from Seasonality: Seasonality repeats within a year. Cyclical patterns take more than 2 years to complete and are not perfectly regular.

2.1.4 Random / Residual (R) — The unexplained noise

What it is: Everything left over after you remove trend, seasonality, and cyclical patterns. This is the unexplained, random variation.

Example: One particular Tuesday, your coffee shop sold 300 coffees instead of the usual 200. Maybe a conference was in town. You couldn’t have predicted it — that extra 100 is “random.”

Key insight: If your model is good, the residuals should look like pure random noise (no pattern left).

2.2 The Decomposition Equation

Once you accept that data is made of these components, you can write a mathematical expression for any time series observation \(y_t\) at time \(t\):

2.2.1 Additive Model

Use this when the seasonal swings stay roughly the same size over time.

\[y_t = T_t + S_t + R_t\]

When to use: The ups and downs in your data look the same height throughout the series (constant variance).

2.2.2 Multiplicative Model

Use this when the seasonal swings get bigger as the trend grows.

\[y_t = T_t \times S_t \times R_t\]

When to use: The wiggling in your data gets wider over time (increasing variance).

Tip: You can convert a multiplicative model into an additive one by taking the logarithm of your data: \[\log(y_t) = \log(T_t) + \log(S_t) + \log(R_t)\]

✏️ Self-check: You’re looking at monthly electricity usage data. In summer and winter the spikes get bigger every year. Should you use an additive or multiplicative model? Why? You should use the multiplicative model because the spikes get bigger every year. Different than the additive model that stays the same size.

3 Classical Decomposition in R

3.1 What decomposition does

Decomposition is the process of separating a time series into its components so you can study each one individually. Think of it like using a prism to split white light into individual colours.

In R, the main tools are:

# Load the forecast library (your main toolbox for this module)
library(forecast)

# Basic decomposition
plot(decompose(my_timeseries, type = "additive"))
plot(decompose(my_timeseries, type = "multiplicative"))

# Better decomposition: STL (Seasonal & Trend using Loess)
stl_result <- stl(my_timeseries, s.window = "periodic")
autoplot(stl_result)

3.2 STL Decomposition — the preferred method

STL stands for Seasonal and Trend decomposition using Loess (Loess is a type of local regression).

Why STL is better than basic decomposition:

  • It handles data at both ends of the series (no missing values at the start or end)
  • It is robust — extreme values (outliers) don’t distort the components
  • You can control how flexible the trend and seasonal components are

The trade-off: STL only works for additive models. For multiplicative data, take logs first, apply STL, then back-transform.

✏️ Self-check: What is the difference between decompose() and stl() in R? Which should you prefer and why? > For decompose() you miss values at the start or the end, outliers distort the components. stl() only works for additive models.

4 Classical Forecasting Models — Exponential Smoothing

4.1 The core idea: recent data matters more

Exponential smoothing is based on a simple but powerful idea: the most recent observations should have more influence on your forecast than older ones.

Imagine you’re estimating how long your commute will take tomorrow. Yesterday’s commute matters more than the commute from 6 months ago. But you don’t completely ignore the older data — you just weight it less.

That’s exactly what exponential smoothing does: it applies exponentially decreasing weights to past observations.

4.2 The three exponential smoothing methods

Think of these as tools with increasing capability:

4.2.1 1. Simple Exponential Smoothing (SES)

Use when: Data has no trend and no seasonality — just a level that fluctuates randomly.

The key parameter — α (alpha):

Alpha value Meaning Practical effect
α close to 1 Short memory Only the very last observation matters
α close to 0 Long memory All past observations weighted equally
α = 0.83 (example) Short memory Forecast closely tracks last value

The formula: \[\hat{y}_{t+1} = \alpha y_t + (1-\alpha)\hat{y}_t\]

This says: “My next forecast = (alpha × what just happened) + (1-alpha × what I previously predicted)”

In R:

library(forecast)

# Method 1: specific SES function
ses_model <- ses(my_timeseries, h = 5)  # h = number of periods to forecast

# Method 2: using the ETS family
ses_model <- ets(my_timeseries, model = "ANN")
# ANN = Error(Additive), No trend, No seasonality

What the forecast looks like: A flat horizontal line — because with no trend or seasonality, the best guess for all future periods is the same value.

4.2.2 2. Holt’s Method (Double Exponential Smoothing)

Use when: Data has a trend but no seasonality.

Two parameters:

  • α (alpha): Controls the level (same as SES)
  • β (beta): Controls how quickly the trend estimate updates

In R:

holt_model <- holt(my_timeseries, h = 5)

# Damped trend (trend slows down over time — often more realistic)
holt_damped <- holt(my_timeseries, damped = TRUE, h = 5)

What the forecast looks like: A sloped line following the trend direction.

4.2.3 3. Holt-Winters Method (Triple Exponential Smoothing)

Use when: Data has both trend and seasonality — the most complete model.

Three parameters:

  • α (alpha): Level
  • β (beta): Trend
  • γ (gamma): Seasonality

Two versions:

  • Additive Holt-Winters: Seasonal effect is constant in size
  • Multiplicative Holt-Winters: Seasonal effect grows with the level

In R:

# Additive
hw_add <- hw(my_timeseries, seasonal = "additive", h = 12)

# Multiplicative
hw_mult <- hw(my_timeseries, seasonal = "multiplicative", h = 12)

# Let R choose automatically using ETS
auto_model <- ets(my_timeseries)
summary(auto_model)

4.3 The ETS Framework — the big family

All exponential smoothing methods belong to the ETS family:

ETS = Error · Trend · Seasonality

Each letter takes a value: - Error: A (additive) or M (multiplicative) - Trend: N (none), A (additive), Ad (additive damped), M (multiplicative) - Seasonality: N (none), A (additive), M (multiplicative)

Model name ETS code Use case
Simple Exp. Smoothing ETS(A,N,N) No trend, no seasonality
Holt’s Method ETS(A,A,N) Trend, no seasonality
Holt-Winters Additive ETS(A,A,A) Trend + additive seasonality
Holt-Winters Multiplicative ETS(A,A,M) Trend + multiplicative seasonality 
# Let R automatically find the best ETS model
best_model <- ets(my_timeseries)
summary(best_model)
forecast(best_model, h = 12)

5 How to Measure Forecast Accuracy

5.1 Why this matters

Any model can give you a forecast. But is it a good forecast? You need numbers to measure how far off your predictions are from reality.

The standard approach:

  1. Split your data into a training set (to build the model) and a test set (to check accuracy)
  2. Forecast the test period using your model
  3. Compare forecasts to actual values using error measures

5.2 The main error measures

Let \(y_t\) = actual value and \(\hat{y}_t\) = forecasted value. The error is \(e_t = y_t - \hat{y}_t\).

Measure Formula What it tells you
MAE (Mean Absolute Error) \(\frac{1}{n}\sum |e_t|\) Average size of errors, ignoring direction
RMSE (Root Mean Square Error) \(\sqrt{\frac{1}{n}\sum e_t^2}\) Penalises large errors more heavily
MAPE (Mean Absolute % Error) \(\frac{100}{n}\sum \left|\frac{e_t}{y_t}\right|\) Error as a percentage (good for comparison)
AIC (Akaike Information Criterion) — Penalises model complexity; lower = better
BIC (Bayesian Information Criterion) — Similar to AIC, stronger complexity penalty

The golden rule for model selection: When comparing two models, the one with the lower AIC/BIC and lower RMSE/MAE is preferred.

# Get accuracy metrics in R
accuracy(forecast_result, test_data)

# Compare AIC between two models
summary(model1)$aic
summary(model2)$aic

✏️ Self-check: You compare two models. Model A has AIC = 1200 and RMSE = 45. Model B has AIC = 1180 and RMSE = 52. Which would you choose and why? (Hint: there’s no single right answer — discuss the trade-off.)

6 ARIMA Models — The Statistical Engine

6.1 The shift in thinking

Everything up to now (exponential smoothing) was about finding patterns and extrapolating them. ARIMA is different — it’s based on statistical theory about how data relates to its own past values.

The key concept: autocorrelation — the idea that today’s value is correlated with yesterday’s value, last week’s value, etc.

Think of it like your mood. If you’re happy today, you’re probably somewhat happy tomorrow. Your mood is correlated with your own past mood. Data often behaves the same way.

6.2 Stationarity — the crucial requirement

Before fitting an ARIMA model, your data must be stationary. This is perhaps the most important concept in the second half of the module.

6.2.1 What does stationary mean?

A time series is stationary if it has three properties that don’t change over time:

  1. Constant mean — the average doesn’t drift up or down
  2. Constant variance — the spread of the data stays roughly the same
  3. Constant covariance — if observation at time 1 is correlated with observation at time 3 (lag 2), then this same correlation exists throughout the entire series

The visual test: If you cover different parts of your time series plot with your hand and each section looks basically the same — same level, same spread — it’s likely stationary.

6.2.2 Why does stationarity matter?

ARIMA models are built on statistical equations that only work when the data’s statistical properties are stable. If the mean is drifting, the model can’t figure out what “normal” is.

6.2.3 How to make data stationary

Problem: non-constant mean (trend) → Solution: Differencing

Differencing means computing the change from one period to the next: \[\Delta y_t = y_t - y_{t-1}\]

If differencing once doesn’t work, you can do it twice (second-order differencing). The number of times you difference is the d in ARIMA(p, d, q).

Problem: non-constant variance → Solution: Log transformation

Taking the log of your data compresses large values and stabilises variance. This is the same reason multiplicative seasonal data responds to log transformation.

# Visual check: does the data look stationary?
plot(my_timeseries)

# Differencing in R
diff_series <- diff(my_timeseries)        # First difference
diff2_series <- diff(my_timeseries, differences = 2)  # Second difference

# Formal statistical test for stationarity (Augmented Dickey-Fuller)
library(tseries)
adf.test(my_timeseries)
# p-value < 0.05 → stationary
# p-value > 0.05 → NOT stationary, need differencing

6.3 The ARIMA building blocks

ARIMA(p, d, q) is made of three parts:

6.3.1 AR(p) — AutoRegressive

The idea: Today’s value depends on its own past values.

\[y_t = \phi_1 y_{t-1} + \phi_2 y_{t-2} + \ldots + \phi_p y_{t-p} + \varepsilon_t\]

Analogy: “My commute today (y_t) is partly explained by my commute yesterday (y_{t-1}) and the day before (y_{t-2}).” The p tells you how many past values to include.

6.3.2 MA(q) — Moving Average

The idea: Today’s value depends on past forecast errors (not past values themselves).

\[y_t = \varepsilon_t + \theta_1 \varepsilon_{t-1} + \theta_2 \varepsilon_{t-2} + \ldots + \theta_q \varepsilon_{t-q}\]

Analogy: “When I was late yesterday, I leave earlier today to compensate — I’m reacting to past errors in my schedule.” The q tells you how many past errors to include.

6.3.3 I(d) — Integrated

The idea: The d stands for how many times you need to difference the data to make it stationary.

  • d = 0: already stationary
  • d = 1: one differencing needed (most common)
  • d = 2: two differences needed (rare)

6.3.4 Putting it together: ARIMA(p, d, q)

ARIMA(p, d, q)
       |  |  |
       |  |  └── q: how many past ERRORS influence today
       |  └───── d: how many times to DIFFERENCE the data
       └──────── p: how many past VALUES influence today
# Fit a specific ARIMA model
arima_model <- arima(my_timeseries, order = c(1, 1, 1))  # ARIMA(1,1,1)

# Let R automatically choose the best ARIMA model
library(forecast)
auto_model <- auto.arima(my_timeseries)
summary(auto_model)
forecast(auto_model, h = 12)

6.4 How to identify p and q: ACF and PACF

The ACF (AutoCorrelation Function) and PACF (Partial AutoCorrelation Function) are visual tools that help you choose p and q.

6.4.1 ACF — AutoCorrelation Function

Shows the correlation between the series and its own past values at each lag.

Reading the ACF plot: - Each bar shows correlation at that lag - Dashed lines = significance boundaries (95% confidence) - Bars beyond the dashed lines = significant correlation at that lag

6.4.2 PACF — Partial AutoCorrelation Function

Shows the correlation with a specific lag after removing the influence of all shorter lags.

6.4.3 The identification rules (the patterns to memorise)

Pattern Meaning Model
ACF: slow decay; PACF: cuts off after lag p Autoregressive AR(p)
ACF: cuts off after lag q; PACF: slow decay Moving average MA(q)
Both decay slowly Mixed ARMA(p,q) 
# Plot ACF and PACF
acf(my_stationary_series, lag.max = 30)   # AutoCorrelation Function
pacf(my_stationary_series, lag.max = 30)  # Partial AutoCorrelation Function

✏️ Self-check: What is the difference between ACF and PACF? If the PACF cuts off sharply after lag 2 and the ACF decays slowly, what model would you suggest?

7 Seasonal ARIMA — SARIMA

7.1 Extending ARIMA to handle seasonality

When your data has both regular autocorrelation AND seasonal patterns, you need SARIMA:

\[\text{SARIMA}(p, d, q)(P, D, Q)_m\]

The lowercase letters (p, d, q) handle the regular (non-seasonal) part — exactly as in ARIMA.

The uppercase letters (P, D, Q) handle the seasonal part, repeating at period m (e.g., m = 12 for monthly data with annual seasonality).

Parameter What it does
P Seasonal AR — correlation with observations m periods ago
D Seasonal differencing — remove seasonal trend
Q Seasonal MA — correlation with seasonal errors
m Season length (12 = monthly, 4 = quarterly) 
# Automatic SARIMA selection
library(forecast)
sarima_model <- auto.arima(my_timeseries, seasonal = TRUE)
summary(sarima_model)

# Manual SARIMA specification
sarima_manual <- arima(my_timeseries, 
                        order = c(1,1,1),          # non-seasonal part
                        seasonal = list(order = c(1,1,1), period = 12))  # seasonal part

8 Residual Diagnostics — Is Your Model Good?

8.1 Why check residuals?

After fitting any model, you must check the residuals (the leftovers). If the model has done its job, the residuals should contain no pattern — they should look like random white noise.

Think of it like this: after a good model has explained everything it can, the only thing left should be unexplainable randomness. If you can still see a pattern in the residuals, your model is missing something.

8.2 The four residual checks

# Plot residuals in four ways
par(mfrow = c(2,2))

# 1. Histogram - should look like a bell curve (normal distribution)
hist(residuals(my_model))

# 2. Boxplot - should be symmetrical around zero
boxplot(residuals(my_model))

# 3. Q-Q Plot - points should follow the diagonal line
qqnorm(residuals(my_model))
qqline(residuals(my_model), col = "red")

# 4. Residuals over time - should look random, no pattern
plot(residuals(my_model))
abline(h = 0, col = "red")

8.2.1 What you’re looking for in each plot

Plot Good sign Bad sign
Histogram Symmetric bell shape Skewed, heavy tails
Boxplot Symmetric, median near zero Asymmetric, many outliers
Q-Q Plot Points on the diagonal line Points curve away from line
Time plot Random scatter, no pattern Visible trend or waves

8.3 The Ljung-Box Test

A formal statistical test for whether residuals are truly random:

Box.test(residuals(my_model), lag = 20, type = "Ljung-Box")
# p-value > 0.05 → residuals are random (good!)
# p-value < 0.05 → residuals still have structure (bad — model needs improvement)

✏️ Self-check: You fit an ARIMA model and plot the residuals. The Q-Q plot shows points curving off the line at both ends. What does this tell you? Is your model acceptable?

9 The Complete Time Series Workflow

9.1 Your step-by-step process for any dataset

This is the workflow you will use in your project and exams. Memorise this sequence:

Step 1: LOOK at the data
        → Plot the time series
        → What do you see? Trend? Seasonality? Growing variance?
        → Is it additive or multiplicative?

Step 2: DECOMPOSE the data
        → Use STL decomposition
        → Examine each component separately
        → Is the seasonal component stable?

Step 3: CHECK stationarity (for ARIMA)
        → Plot ACF of original data — does it decay slowly? (non-stationary)
        → Run ADF test
        → Apply transformations if needed (log, differencing)

Step 4: FIT models
        → Try multiple candidate models
        → For classical: SES / Holt / Holt-Winters
        → For stochastic: ARIMA / SARIMA
        → Use auto.arima() as a starting point

Step 5: COMPARE models
        → Compare AIC, BIC, RMSE, MAE
        → Lower = better

Step 6: DIAGNOSE residuals
        → Check histogram, Q-Q plot, time plot
        → Run Ljung-Box test
        → If residuals have pattern → go back to Step 4

Step 7: FORECAST
        → Generate forecast with confidence intervals
        → Plot and interpret the results

9.2 Quick R workflow template

# ---- Step 1: Load and plot ----
library(forecast)
library(tseries)

ts_data <- ts(my_data, start = c(2000, 1), frequency = 12)
plot(ts_data)

# ---- Step 2: Decompose ----
stl_decomp <- stl(ts_data, s.window = "periodic")
autoplot(stl_decomp)

# ---- Step 3: Check stationarity ----
acf(ts_data)
adf.test(ts_data)

# Difference if needed
ts_diff <- diff(ts_data)
adf.test(ts_diff)

# ---- Step 4: Fit models ----
# Classical
hw_model <- hw(ts_data, seasonal = "multiplicative", h = 12)

# ARIMA
arima_auto <- auto.arima(ts_data, seasonal = TRUE)

# ---- Step 5: Compare ----
accuracy(hw_model)
summary(arima_auto)  # check AIC

# ---- Step 6: Diagnose residuals ----
checkresiduals(arima_auto)
Box.test(residuals(arima_auto), lag = 20, type = "Ljung-Box")

# ---- Step 7: Forecast ----
fc <- forecast(arima_auto, h = 12)
autoplot(fc)

10 How Concepts Connect Across Subjects

10.1 Links to regression analysis (your other module)

Time series and regression share the same underlying goal: explain variation in an outcome variable. The key differences are:

Regression Time Series
Data points are independent Data points are correlated over time
Predict using other variables (X → Y) Predict using own past values (Y_past → Y_future)
Residuals should be independent Residuals should be random (no autocorrelation)

The bridge concept — regression with time: Your professor mentioned that you can use regression techniques on time series data. This is called a causal time series model — where you use both the series’ own history AND external variables (like weather, income) to explain the data.

10.2 Links to statistics fundamentals

Concept you know How it appears in time series
Normal distribution Residuals should follow N(0, σ²)
Hypothesis testing ADF test for stationarity; Ljung-Box test for residuals
Confidence intervals Forecast intervals widen as you predict further
Minimising sum of squared errors How α, β, γ parameters are estimated
Model selection (R², AIC) AIC/BIC used to choose between models

11 Quick Reference Summary

11.1 Model selection cheat sheet

Look at your data:
├── No trend, no seasonality → SES / ARIMA(p,0,q)
├── Trend only → Holt's / ARIMA(p,1,q) 
├── Trend + seasonality, constant swings → Holt-Winters Additive / SARIMA
└── Trend + seasonality, growing swings → Holt-Winters Multiplicative (or log first)

11.2 Key terms glossary

Term Plain English definition
Time series Data recorded at regular intervals over time
Trend Long-term upward or downward direction
Seasonality Predictable repeating pattern within a year
Cyclical Irregular long-term waves (>2 years)

# What is Time Series Analysis? The Big Picture

## The core idea in one sentence

**Time series analysis is the art of listening to data over time,
understanding its patterns, and using those patterns to predict the
future.**

## A real-world analogy to make it stick

Imagine you work at a coffee shop and every day you record how many
coffees you sell. After a year, you notice:

-   Sales have been **growing overall** (more customers each month) →
    this is **Trend**
-   Sales **spike every Monday morning** and dip on Sundays → this is
    **Seasonality**
-   One week sales dropped because of a snowstorm → this is **Random
    noise**

Time series analysis is *exactly* this: you look at your coffee sales
data, pull apart these different layers, understand what drives each
one, and then use that understanding to say "next month we'll probably
sell around X coffees."

## Why does it matter beyond this module?

This thinking transfers to **any subject or job** that involves data
collected over time:

|-----------------|-----------------------------------------|
| Finance         | Predicting stock prices, exchange rates |
| Healthcare      | Tracking disease spread over weeks      |
| Marketing       | Forecasting monthly sales               |
| Engineering     | Monitoring machine performance          |
| Economics       | Modelling GDP, inflation, unemployment  |
| Climate Science | Analysing temperature changes           |

The concepts you learn here — patterns, prediction, uncertainty — are
universal.

# The Four Components of Any Time Series

## The recipe analogy

Think of a time series like a recipe. A cake is made of flour, sugar,
butter, and eggs — but you only see the finished cake. Time series
analysis is like *reverse-engineering* a recipe: you have the finished
cake (your data), and you want to figure out how much of each ingredient
went in.

The "ingredients" of any time series are:

### Trend (T) — The long-term direction

**What it is:** The overall movement of your data upwards or downwards
over a long period.

**Example:** Global average temperatures have been rising for 100 years.
If you plot this, the long-term *direction* is upward — that upward
slope is the trend.

**Key insight:** A trend doesn't have to be a straight line. It can
curve, slow down, or reverse.

### Seasonality (S) — The repeating pattern

**What it is:** A regular, predictable pattern that repeats within a
fixed period (usually within one year).

**Example:** Ice cream sales are always high in summer and low in winter
— every single year, without fail. This is seasonal behaviour.

**Key insight:** Seasonality is *predictable and regular*. It completes
itself within a calendar year.

### Cyclical Component (C) — The long, irregular wave

**What it is:** Patterns that repeat over multiple years, but are not as
regular as seasons. Usually linked to economic cycles.

**Example:** After the 2009 financial crisis, nearly every economic
indicator dropped. This is a cyclical downturn — it lasts years, not
months, and cannot be predicted with the same precision as seasons.

**Key difference from Seasonality:** Seasonality repeats within a year.
Cyclical patterns take *more than 2 years* to complete and are not
perfectly regular.

### Random / Residual (R) — The unexplained noise

**What it is:** Everything left over after you remove trend,
seasonality, and cyclical patterns. This is the unexplained, random
variation.

**Example:** One particular Tuesday, your coffee shop sold 300 coffees
instead of the usual 200. Maybe a conference was in town. You couldn't
have predicted it — that extra 100 is "random."

**Key insight:** If your model is good, the residuals should look like
pure random noise (no pattern left).

------------------------------------------------------------------------

## The Decomposition Equation

Once you accept that data is made of these components, you can write a
mathematical expression for any time series observation $y_t$ at time
$t$:

### Additive Model

Use this when the seasonal swings stay roughly **the same size** over
time.

$$y_t = T_t + S_t + R_t$$

**When to use:** The ups and downs in your data look the same height
throughout the series (constant variance).

### Multiplicative Model

Use this when the seasonal swings get **bigger as the trend grows**.

$$y_t = T_t \times S_t \times R_t$$

**When to use:** The wiggling in your data gets wider over time
(increasing variance).

**Tip:** You can convert a multiplicative model into an additive one by
taking the logarithm of your data:
$$\log(y_t) = \log(T_t) + \log(S_t) + \log(R_t)$$

# Classical Decomposition in R

## What decomposition does

Decomposition is the process of separating a time series into its
components so you can study each one individually. Think of it like
using a prism to split white light into individual colours.

In R, the main tools are:

``` r
# Load the forecast library (your main toolbox for this module)
library(forecast)

# Basic decomposition
plot(decompose(my_timeseries, type = "additive"))
plot(decompose(my_timeseries, type = "multiplicative"))

# Better decomposition: STL (Seasonal & Trend using Loess)
stl_result <- stl(my_timeseries, s.window = "periodic")
autoplot(stl_result)
```

## STL Decomposition — the preferred method

STL stands for **Seasonal and Trend decomposition using Loess** (Loess
is a type of local regression).

**Why STL is better than basic decomposition:**

-   It handles data at both ends of the series (no missing values at the
    start or end)
-   It is **robust** — extreme values (outliers) don't distort the
    components
-   You can control how flexible the trend and seasonal components are

**The trade-off:** STL only works for **additive** models. For
multiplicative data, take logs first, apply STL, then back-transform.

# Classical Forecasting Models — Exponential Smoothing

## The core idea: recent data matters more

Exponential smoothing is based on a simple but powerful idea: **the most
recent observations should have more influence on your forecast than
older ones.**

Imagine you're estimating how long your commute will take tomorrow.
Yesterday's commute matters more than the commute from 6 months ago. But
you don't completely ignore the older data — you just weight it less.

That's exactly what exponential smoothing does: it applies
*exponentially decreasing weights* to past observations.

## The three exponential smoothing methods

Think of these as tools with increasing capability:

### 1. Simple Exponential Smoothing (SES)

**Use when:** Data has **no trend** and **no seasonality** — just a
level that fluctuates randomly.

**The key parameter — α (alpha):**

| Alpha value        | Meaning      | Practical effect                       |
|--------------------|--------------|----------------------------------------|
| α close to 1       | Short memory | Only the very last observation matters |
| α close to 0       | Long memory  | All past observations weighted equally |
| α = 0.83 (example) | Short memory | Forecast closely tracks last value     |

**The formula:** $$\hat{y}_{t+1} = \alpha y_t + (1-\alpha)\hat{y}_t$$

This says: "My next forecast = (alpha × what just happened) + (1-alpha ×
what I previously predicted)"

**In R:**

``` r
library(forecast)

# Method 1: specific SES function
ses_model <- ses(my_timeseries, h = 5)  # h = number of periods to forecast

# Method 2: using the ETS family
ses_model <- ets(my_timeseries, model = "ANN")
# ANN = Error(Additive), No trend, No seasonality
```

**What the forecast looks like:** A flat horizontal line — because with
no trend or seasonality, the best guess for all future periods is the
same value.

------------------------------------------------------------------------

### 2. Holt's Method (Double Exponential Smoothing)

**Use when:** Data has a **trend** but **no seasonality**.

**Two parameters:**

-   **α (alpha):** Controls the level (same as SES)
-   **β (beta):** Controls how quickly the trend estimate updates

**In R:**

``` r
holt_model <- holt(my_timeseries, h = 5)

# Damped trend (trend slows down over time — often more realistic)
holt_damped <- holt(my_timeseries, damped = TRUE, h = 5)
```

**What the forecast looks like:** A sloped line following the trend
direction.

------------------------------------------------------------------------

### 3. Holt-Winters Method (Triple Exponential Smoothing)

**Use when:** Data has both **trend** and **seasonality** — the most
complete model.

**Three parameters:**

-   **α (alpha):** Level
-   **β (beta):** Trend
-   **γ (gamma):** Seasonality

**Two versions:**

-   **Additive Holt-Winters:** Seasonal effect is constant in size
-   **Multiplicative Holt-Winters:** Seasonal effect grows with the
    level

**In R:**

``` r
# Additive
hw_add <- hw(my_timeseries, seasonal = "additive", h = 12)

# Multiplicative
hw_mult <- hw(my_timeseries, seasonal = "multiplicative", h = 12)

# Let R choose automatically using ETS
auto_model <- ets(my_timeseries)
summary(auto_model)
```

------------------------------------------------------------------------

## The ETS Framework — the big family

All exponential smoothing methods belong to the **ETS** family:

**ETS = Error · Trend · Seasonality**

Each letter takes a value: - **Error:** A (additive) or M
(multiplicative) - **Trend:** N (none), A (additive), Ad (additive
damped), M (multiplicative) - **Seasonality:** N (none), A (additive), M
(multiplicative)

| Model name                  | ETS code   | Use case                           |
|------------------------|------------------------|------------------------|
| Simple Exp. Smoothing       | ETS(A,N,N) | No trend, no seasonality           |
| Holt's Method               | ETS(A,A,N) | Trend, no seasonality              |
| Holt-Winters Additive       | ETS(A,A,A) | Trend + additive seasonality       |
| Holt-Winters Multiplicative | ETS(A,A,M) | Trend + multiplicative seasonality |

``` r
# Let R automatically find the best ETS model
best_model <- ets(my_timeseries)
summary(best_model)
forecast(best_model, h = 12)
```

# How to Measure Forecast Accuracy

## Why this matters

Any model can give you *a* forecast. But is it a *good* forecast? You
need numbers to measure how far off your predictions are from reality.

The standard approach:

1.  **Split your data** into a training set (to build the model) and a
    test set (to check accuracy)
2.  **Forecast** the test period using your model
3.  **Compare** forecasts to actual values using error measures

## The main error measures

Let $y_t$ = actual value and $\hat{y}_t$ = forecasted value. The error
is $e_t = y_t - \hat{y}_t$.

| Measure | Formula | What it tells you |
|------------------------|------------------------|------------------------|
| **MAE** (Mean Absolute Error) | $\frac{1}{n}\sum |e_t|$ | Average size of errors, ignoring direction |
| **RMSE** (Root Mean Square Error) | $\sqrt{\frac{1}{n}\sum e_t^2}$ | Penalises large errors more heavily |
| **MAPE** (Mean Absolute % Error) | $\frac{100}{n}\sum \left|\frac{e_t}{y_t}\right|$ | Error as a percentage (good for comparison) |
| **AIC** (Akaike Information Criterion) | — | Penalises model complexity; **lower = better** |
| **BIC** (Bayesian Information Criterion) | — | Similar to AIC, stronger complexity penalty |

**The golden rule for model selection:** When comparing two models, the
one with the **lower AIC/BIC and lower RMSE/MAE** is preferred.

``` r
# Get accuracy metrics in R
accuracy(forecast_result, test_data)

# Compare AIC between two models
summary(model1)$aic
summary(model2)$aic
```

# ARIMA Models — The Statistical Engine

## The shift in thinking

Everything up to now (exponential smoothing) was about finding patterns
and extrapolating them. ARIMA is different — it's based on **statistical
theory** about how data relates to its own past values.

The key concept: **autocorrelation** — the idea that today's value is
correlated with yesterday's value, last week's value, etc.

Think of it like your mood. If you're happy today, you're probably
somewhat happy tomorrow. Your mood is correlated with your own past
mood. Data often behaves the same way.

## Stationarity — the crucial requirement

Before fitting an ARIMA model, your data must be **stationary**. This is
perhaps the most important concept in the second half of the module.

### What does stationary mean?

A time series is stationary if it has **three properties that don't
change over time:**

1.  **Constant mean** — the average doesn't drift up or down
2.  **Constant variance** — the spread of the data stays roughly the
    same
3.  **Constant covariance** — if observation at time 1 is correlated
    with observation at time 3 (lag 2), then this same correlation
    exists throughout the entire series

**The visual test:** If you cover different parts of your time series
plot with your hand and each section looks basically the same — same
level, same spread — it's likely stationary.

### Why does stationarity matter?

ARIMA models are built on statistical equations that only work when the
data's statistical properties are stable. If the mean is drifting, the
model can't figure out what "normal" is.

### How to make data stationary

**Problem: non-constant mean (trend)** → Solution: **Differencing**

Differencing means computing the change from one period to the next:
$$\Delta y_t = y_t - y_{t-1}$$

If differencing once doesn't work, you can do it twice (second-order
differencing). The number of times you difference is the **d** in
ARIMA(p, d, q).

**Problem: non-constant variance** → Solution: **Log transformation**

Taking the log of your data compresses large values and stabilises
variance. This is the same reason multiplicative seasonal data responds
to log transformation.

``` r
# Visual check: does the data look stationary?
plot(my_timeseries)

# Differencing in R
diff_series <- diff(my_timeseries)        # First difference
diff2_series <- diff(my_timeseries, differences = 2)  # Second difference

# Formal statistical test for stationarity (Augmented Dickey-Fuller)
library(tseries)
adf.test(my_timeseries)
# p-value < 0.05 → stationary
# p-value > 0.05 → NOT stationary, need differencing
```

------------------------------------------------------------------------

## The ARIMA building blocks

**ARIMA(p, d, q)** is made of three parts:

### AR(p) — AutoRegressive

**The idea:** Today's value depends on its own past values.

$$y_t = \phi_1 y_{t-1} + \phi_2 y_{t-2} + \ldots + \phi_p y_{t-p} + \varepsilon_t$$

**Analogy:** "My commute today (y_t) is partly explained by my commute
yesterday (y\_{t-1}) and the day before (y\_{t-2})." The **p** tells you
how many past values to include.

### MA(q) — Moving Average

**The idea:** Today's value depends on past *forecast errors* (not past
values themselves).

$$y_t = \varepsilon_t + \theta_1 \varepsilon_{t-1} + \theta_2 \varepsilon_{t-2} + \ldots + \theta_q \varepsilon_{t-q}$$

**Analogy:** "When I was late yesterday, I leave earlier today to
compensate — I'm reacting to past *errors* in my schedule." The **q**
tells you how many past errors to include.

### I(d) — Integrated

**The idea:** The d stands for how many times you need to *difference*
the data to make it stationary.

-   d = 0: already stationary
-   d = 1: one differencing needed (most common)
-   d = 2: two differences needed (rare)

### Putting it together: ARIMA(p, d, q)

```         
ARIMA(p, d, q)
       |  |  |
       |  |  └── q: how many past ERRORS influence today
       |  └───── d: how many times to DIFFERENCE the data
       └──────── p: how many past VALUES influence today
```

``` r
# Fit a specific ARIMA model
arima_model <- arima(my_timeseries, order = c(1, 1, 1))  # ARIMA(1,1,1)

# Let R automatically choose the best ARIMA model
library(forecast)
auto_model <- auto.arima(my_timeseries)
summary(auto_model)
forecast(auto_model, h = 12)
```

------------------------------------------------------------------------

## How to identify p and q: ACF and PACF

The **ACF** (AutoCorrelation Function) and **PACF** (Partial
AutoCorrelation Function) are visual tools that help you choose p and q.

### ACF — AutoCorrelation Function

Shows the correlation between the series and its own past values at each
lag.

**Reading the ACF plot:** - Each bar shows correlation at that lag -
Dashed lines = significance boundaries (95% confidence) - Bars beyond
the dashed lines = *significant* correlation at that lag

### PACF — Partial AutoCorrelation Function

Shows the correlation with a specific lag *after removing the influence
of all shorter lags*.

### The identification rules (the patterns to memorise)

| Pattern                                     | Meaning        | Model     |
|---------------------------------------------|----------------|-----------|
| ACF: slow decay; PACF: cuts off after lag p | Autoregressive | AR(p)     |
| ACF: cuts off after lag q; PACF: slow decay | Moving average | MA(q)     |
| Both decay slowly                           | Mixed          | ARMA(p,q) |

``` r
# Plot ACF and PACF
acf(my_stationary_series, lag.max = 30)   # AutoCorrelation Function
pacf(my_stationary_series, lag.max = 30)  # Partial AutoCorrelation Function
```

# Seasonal ARIMA — SARIMA

## Extending ARIMA to handle seasonality

When your data has both regular autocorrelation AND seasonal patterns,
you need **SARIMA**:

$$\text{SARIMA}(p, d, q)(P, D, Q)_m$$

The lowercase letters (p, d, q) handle the regular (non-seasonal) part —
exactly as in ARIMA.

The uppercase letters (P, D, Q) handle the **seasonal** part, repeating
at period **m** (e.g., m = 12 for monthly data with annual seasonality).

| Parameter | What it does                                              |
|-----------|-----------------------------------------------------------|
| P         | Seasonal AR — correlation with observations m periods ago |
| D         | Seasonal differencing — remove seasonal trend             |
| Q         | Seasonal MA — correlation with seasonal errors            |
| m         | Season length (12 = monthly, 4 = quarterly)               |

``` r
# Automatic SARIMA selection
library(forecast)
sarima_model <- auto.arima(my_timeseries, seasonal = TRUE)
summary(sarima_model)

# Manual SARIMA specification
sarima_manual <- arima(my_timeseries, 
                        order = c(1,1,1),          # non-seasonal part
                        seasonal = list(order = c(1,1,1), period = 12))  # seasonal part
```

# Residual Diagnostics — Is Your Model Good?

## Why check residuals?

After fitting any model, you must check the residuals (the leftovers).
If the model has done its job, the residuals should contain **no
pattern** — they should look like random white noise.

Think of it like this: after a good model has explained everything it
can, the only thing left should be unexplainable randomness. If you can
still see a pattern in the residuals, your model is missing something.

## The four residual checks

``` r
# Plot residuals in four ways
par(mfrow = c(2,2))

# 1. Histogram - should look like a bell curve (normal distribution)
hist(residuals(my_model))

# 2. Boxplot - should be symmetrical around zero
boxplot(residuals(my_model))

# 3. Q-Q Plot - points should follow the diagonal line
qqnorm(residuals(my_model))
qqline(residuals(my_model), col = "red")

# 4. Residuals over time - should look random, no pattern
plot(residuals(my_model))
abline(h = 0, col = "red")
```

### What you're looking for in each plot

| Plot      | Good sign                   | Bad sign                    |
|-----------|-----------------------------|-----------------------------|
| Histogram | Symmetric bell shape        | Skewed, heavy tails         |
| Boxplot   | Symmetric, median near zero | Asymmetric, many outliers   |
| Q-Q Plot  | Points on the diagonal line | Points curve away from line |
| Time plot | Random scatter, no pattern  | Visible trend or waves      |

## The Ljung-Box Test

A formal statistical test for whether residuals are truly random:

``` r
Box.test(residuals(my_model), lag = 20, type = "Ljung-Box")
# p-value > 0.05 → residuals are random (good!)
# p-value < 0.05 → residuals still have structure (bad — model needs improvement)
```

# The Complete Time Series Workflow

## Your step-by-step process for any dataset

This is the workflow you will use in your project and exams. Memorise
this sequence:

```         
Step 1: LOOK at the data
        → Plot the time series
        → What do you see? Trend? Seasonality? Growing variance?
        → Is it additive or multiplicative?

Step 2: DECOMPOSE the data
        → Use STL decomposition
        → Examine each component separately
        → Is the seasonal component stable?

Step 3: CHECK stationarity (for ARIMA)
        → Plot ACF of original data — does it decay slowly? (non-stationary)
        → Run ADF test
        → Apply transformations if needed (log, differencing)

Step 4: FIT models
        → Try multiple candidate models
        → For classical: SES / Holt / Holt-Winters
        → For stochastic: ARIMA / SARIMA
        → Use auto.arima() as a starting point

Step 5: COMPARE models
        → Compare AIC, BIC, RMSE, MAE
        → Lower = better

Step 6: DIAGNOSE residuals
        → Check histogram, Q-Q plot, time plot
        → Run Ljung-Box test
        → If residuals have pattern → go back to Step 4

Step 7: FORECAST
        → Generate forecast with confidence intervals
        → Plot and interpret the results
```

## Quick R workflow template

``` r
# ---- Step 1: Load and plot ----
library(forecast)
library(tseries)

ts_data <- ts(my_data, start = c(2000, 1), frequency = 12)
plot(ts_data)

# ---- Step 2: Decompose ----
stl_decomp <- stl(ts_data, s.window = "periodic")
autoplot(stl_decomp)

# ---- Step 3: Check stationarity ----
acf(ts_data)
adf.test(ts_data)

# Difference if needed
ts_diff <- diff(ts_data)
adf.test(ts_diff)

# ---- Step 4: Fit models ----
# Classical
hw_model <- hw(ts_data, seasonal = "multiplicative", h = 12)

# ARIMA
arima_auto <- auto.arima(ts_data, seasonal = TRUE)

# ---- Step 5: Compare ----
accuracy(hw_model)
summary(arima_auto)  # check AIC

# ---- Step 6: Diagnose residuals ----
checkresiduals(arima_auto)
Box.test(residuals(arima_auto), lag = 20, type = "Ljung-Box")

# ---- Step 7: Forecast ----
fc <- forecast(arima_auto, h = 12)
autoplot(fc)
```

# How Concepts Connect Across Subjects

## Links to regression analysis (your other module)

Time series and regression share the same underlying goal: **explain
variation in an outcome variable**. The key differences are:

| Regression | Time Series |
|------------------------------------|------------------------------------|
| Data points are independent | Data points are correlated over time |
| Predict using other variables (X → Y) | Predict using own past values (Y_past → Y_future) |
| Residuals should be independent | Residuals should be random (no autocorrelation) |

**The bridge concept — regression with time:** Your professor mentioned
that you can use regression techniques on time series data. This is
called a **causal time series model** — where you use both the series'
own history AND external variables (like weather, income) to explain the
data.

## Links to statistics fundamentals

| Concept you know | How it appears in time series |
|------------------------------------|------------------------------------|
| Normal distribution | Residuals should follow N(0, σ²) |
| Hypothesis testing | ADF test for stationarity; Ljung-Box test for residuals |
| Confidence intervals | Forecast intervals widen as you predict further |
| Minimising sum of squared errors | How α, β, γ parameters are estimated |
| Model selection (R², AIC) | AIC/BIC used to choose between models |

| **Multiplicative model** | Components multiplied; seasonal size grows with level |
| **Stationary** | Data with constant mean, variance, and covariance over time |
| **Differencing** | Subtracting consecutive values to remove trend |
| **ACF** | How correlated a series is with its own past values |
| **PACF** | Direct correlation at each lag (removing shorter lag effects) |
| **α (alpha)** | Smoothing parameter for the level (0–1) |
| **β (beta)** | Smoothing parameter for the trend |
| **γ (gamma)** | Smoothing parameter for the seasonality |
| **AIC/BIC** | Model quality scores — lower means better |
| **ARIMA(p,d,q)** | Autoregressive + Differencing + Moving Average model |
| **SARIMA** | ARIMA extended for seasonal data |
