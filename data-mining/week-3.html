3 The Titanic Dataset

3.1 What is it?

The Titanic dataset is one of the most well-known datasets in machine learning. It records information about passengers aboard the RMS Titanic, which sank in April 1912. The central question is:

Can we build a model that predicts which passengers survived the Titanic disaster?

This is a binary classification problem — for each passenger, the model must predict one of two outcomes: survived (1) or did not survive (0).

3.2 Variables in the Dataset

Variable Description Notes
PassengerId Unique ID for each passenger Not useful for prediction
Survived 0 = No, 1 = Yes Target variable
Pclass Ticket class: 1 = 1st, 2 = 2nd, 3 = 3rd Very important predictor
Name Passenger’s full name Unique — drop it
Sex Gender Very strong predictor
Age Age in years Has many missing values
SibSp # of siblings/spouses aboard Combined with Parch
Parch # of parents/children aboard Combined with SibSp
Ticket Ticket number Likely not useful as-is
Fare Price paid for the ticket Correlated with Pclass
Cabin Cabin number Lots of missing values
Embarked Port of embarkation: C, Q, or S Some predictive signal

4 Data Exploration

4.1 Why Explore Data Before Modelling?

Data exploration is a critical step before building any machine learning model. It helps you:

  • Understand what the data looks like
  • Identify missing values and decide how to handle them
  • Spot variables that are likely to be useful predictors
  • Detect relationships between predictor variables (not just between each variable and the target)
  • Avoid mistakes that would silently damage your model’s quality

Rule of thumb: Always know your data before you model it. A model built on poorly understood data will produce unreliable results.

4.2 Checking Unique Values and Missing Data

# Count unique values per column
uniq_len <- function(x) { length(unique(x)) }
sapply(titanic_train, uniq_len)

# Count missing values (NAs or empty strings)
countBlank <- function(x) { sum(is.na(x) | x == "") }
sapply(titanic_train, countBlank)

What to look for:

  • Columns with as many unique values as rows (e.g. PassengerId, Name) → These are IDs, not features. Drop them.
  • Columns with too many missing values (e.g. Cabin has 687 missing out of 891) → Consider dropping or imputing.
  • Unexpected number of unique values (e.g. Embarked shows 4 unique values when only 3 are expected) → Signals there are blank/null entries that need fixing.

4.3 Key Findings from the Titanic Data

4.3.1 Family Size vs. Survival

A new variable family is created by adding SibSp + Parch to represent total family members travelling together.

titanic_train$family <- titanic_train$SibSp + titanic_train$Parch
titanic_train$Survived <- factor(titanic_train$Survived)

ggplot(titanic_train, aes(x = family, fill = factor(Survived))) +
  geom_bar(stat = 'count', position = 'dodge') +
  labs(x = 'Family Size') +
  ggtitle("Family size v Survived")

Finding: Solo travellers had the lowest survival rate. Passengers in small family groups (1–3 others) had noticeably higher survival rates. Very large families (6+) had very few records, so conclusions there are not reliable.

Interpretation: Being in a family likely means children were present, and children were given priority in lifeboats.

4.3.2 Ticket Class vs. Survival

p01 <- ggplot(titanic_train, aes(Pclass, fill = Survived)) +
  geom_bar(position = "stack") + ggtitle("By counts")

p02 <- ggplot(titanic_train, aes(Pclass, fill = Survived)) +
  geom_bar(position = "fill") +
  scale_y_continuous(labels = scales::percent) + ggtitle("By proportion")

gridExtra::grid.arrange(p01, p02, nrow = 1, top = 'Survived by Class')

Finding:

  • 1st class: ~63% survived
  • 2nd class: ~47% survived
  • 3rd class: ~24% survived

Important tip — plot both counts AND proportions:

  • Counts tell you how many passengers are in each group (the effect size)
  • Proportions tell you how different the survival rate is from the baseline of 38% (the signal strength)

Both matter. A strong signal (e.g. 90% survival) in a group with only 10 records is not very useful — it barely affects the overall model.

4.3.3 Gender vs. Survival

Finding:

  • Female passengers: ~74% survived
  • Male passengers: ~19% survived

Gender is the single strongest predictor in this dataset. This aligns with the historical “women and children first” policy used when loading lifeboats.

4.3.4 Fixing the Embarked Variable

Two passengers had blank values for Embarked. Since Southampton was by far the most common embarkation port, those blanks were reassigned to S.

titanic_train[titanic_train$Embarked == "", ]$Embarked <- 'S'

Why not create a new “Unknown” category? Because with only 2 records, assigning them a unique value X could cause the model to over-fit on that tiny group. The safer choice is to merge them into the most common category.

4.3.5 Fare vs. Survival

A box plot of Fare by Survived shows that passengers who paid higher fares had a somewhat higher survival rate. However, fare is closely correlated with ticket class — they carry overlapping information.

4.3.6 Age vs. Survival — Creating a Derived Variable

A simple box plot of Age vs. Survived doesn’t show a strong difference. This is because the real signal is not about age in general, but specifically about children.

temp <- titanic_train[!is.na(titanic_train$Age), ]
temp$is_child <- ifelse(temp$Age < 17, 1, 0)

Children (under 17) had roughly a 50% survival rate vs. about 35% for adults. Creating a binary is_child variable helps the machine learning algorithm pick up on this threshold effect.

Key concept — derived variables: Sometimes the raw variable is not as useful as a transformed or binned version of it. Think about what threshold or transformation would best capture the relationship with the target.

4.3.7 Pairs Plot

columns <- c("Pclass", "Age", "Fare", "family")
pairs.panels(titanic_train[, columns], method = "pearson",
             hist.col = "#00AFBB", density = TRUE, ellipses = FALSE)

Finding: The strongest correlation among predictors is between Pclass and Fare (–0.55), which makes sense — first class passengers paid much more.

Why this matters: If two predictor variables are highly correlated, they carry redundant information. The model may just pick one of them and largely ignore the other.

4.3.8 Embarkation and Ticket Class

A deeper investigation reveals why passengers from Cherbourg (C) had a higher survival rate: a much higher proportion of first-class passengers boarded at Cherbourg compared to Queenstown or Southampton. The survival signal from Embarked is largely already captured by Pclass.

This is an example of a variable that appears predictive but is actually a proxy for another variable. Including such variables in a model can be misleading.

5 Supervised Learning — Classification

5.1 What is it?

Supervised learning means training a model on data where we already know the correct answers (the labels), so the model learns to predict those labels for new, unseen data.

Classification is a type of supervised learning where the output is a category (e.g. survived / not survived, spam / not spam, cat / dog).

  • Binary classification: two possible outcomes (like Titanic survived yes/no)
  • Multi-class classification: three or more outcomes (like Iris species: setosa / versicolor / virginica)

5.2 Common Classification Algorithms

Algorithm Key Characteristic
Decision Trees Simple, interpretable, easy to visualise
Logistic Regression Probabilistic, good baseline model
Support Vector Machines Strong with high-dimensional data
Naive Bayes Fast, works well with text
Random Forests Ensemble of trees, very powerful
Gradient Boosted Trees Often best on tabular data
Deep Learning Best for images, text, audio

6 Building a Classification Model in R

6.1 The Core Workflow

library(rpart)
data(iris)

# 1. Set a random seed for reproducibility
set.seed(1987)

# 2. Split data into training (90%) and test (10%)
nobs <- nrow(iris)
train <- sample(nobs, 0.9 * nobs)
test <- setdiff(seq_len(nobs), train)

# 3. Train the model on training data ONLY
rpart_model <- rpart(Species ~ ., data = iris[train, ], method = "class")

# 4. Get predictions on the test set
pr <- predict(rpart_model, newdata = iris[test, ], type = "class")

# 5. Evaluate performance
tab <- table(pr, iris[test, ]$Species, dnn = c("predicted", "actual"))
print(tab)

accuracy <- 100 * sum(diag(tab)) / sum(tab)
print(paste('Model accuracy on test data =', round(accuracy, 2)))

6.2 Why Split Into Train and Test?

A model will always look better on data it was trained on, because it has already “seen” that data. Evaluating on training data is like giving someone an exam with the answer sheet in front of them — it tells you nothing about how well they actually understand the material.

The test set must be:

  • Created before building the model
  • Completely separate from the training set (no overlap)
  • Representative of data the model will see in the real world

Important: The evaluation metrics you report should come from the test set, not the training set.

6.3 Reproducible Data Science

set.seed(1987)  # Always set this before any random operations

Setting a random seed ensures that every time the code is run (on any machine), it produces the exact same train/test split. This is essential for reproducible data science — other people must be able to reproduce your results exactly, or your findings cannot be trusted.

6.4 What is Overfitting / Overtraining?

Overfitting happens when a model learns the training data too well — including its noise and quirks — and fails to generalise to new data.

Example: If the model memorises that the 2 passengers with a blank Embarked value both survived, it will predict survival for any future record with a blank embarkation. This is not a real pattern, just noise — and the model will perform poorly on new data.

6.5 Model Drift and Data Drift

  • Data drift: The distribution of real-world data changes over time, so the model (trained on older data) becomes less accurate.
  • Model drift: The relationship between the input variables and the target variable changes over time.

Both of these mean that machine learning models need to be monitored and periodically retrained.

7 Decision Trees

7.1 What is a Decision Tree?

A decision tree is one of the simplest and most intuitive machine learning algorithms. It works by splitting the data into groups based on a series of yes/no questions (conditions).

Analogy: Imagine a game of 20 questions. Each question splits the possibilities in two, and you keep narrowing down until you reach a final answer.

7.2 How it Works

  1. Start with all the training data at the root node
  2. Find the best attribute (column) and the best split point that separates the data into more “pure” groups
  3. Create two branches (left = condition is true, right = condition is false)
  4. Repeat for each sub-group (node) until:
    • All records in a node belong to the same class (it becomes a leaf node), or
    • There are no more useful attributes to split on

7.3 How to Choose the Best Split?

The algorithm uses a measure of impurity to decide which split is best. The goal is always to reduce impurity — i.e. to create groups that are as “pure” (dominated by a single class) as possible.

Two common impurity measures:

\[\text{Entropy}(t) = -\sum_{i=0}^{c-1} p(i|t) \log_2 p(i|t)\]

\[\text{Gini}(t) = 1 - \sum_{i=0}^{c-1} [p(i|t)]^2\]

Where c is the number of classes and p(i|t) is the probability that a record at node t belongs to class i.

Information Gain is calculated by comparing the impurity of the parent node to the weighted average impurity of the child nodes after the split. The split with the highest information gain is chosen.

Intuition: A node where 50% of records are class 0 and 50% are class 1 is maximally impure (entropy = 1). A node where 100% of records are the same class is perfectly pure (entropy = 0).

7.4 Advantages and Disadvantages

Advantages Disadvantages
Easy to understand and visualise Other algorithms usually outperform it
Can deal with non-linear data Tends to overfit (especially deep trees)
No need to normalise data Sensitive to small changes in data
Can be expressed as if/then rules Does not handle missing data well

7.5 Handling Numeric Data

Since every node is a yes/no condition, numeric variables are handled by splitting on a threshold value (e.g. Age >= 13, Fare >= 23). The algorithm finds the best threshold automatically.

7.6 Pruning

Decision trees can grow very large and complex, overfitting the training data. Pruning means cutting off some of the lower branches to simplify the tree and improve generalisation.

7.7 Example — Titanic Decision Tree

The decision tree built on the Titanic dataset finds the following key splits (starting from the root):

  1. Sex = male? → Most important split. Males generally did not survive.
  2. For males: Age >= 13? → Younger boys had better survival odds
  3. For females: Pclass >= 3? → 3rd class females had lower survival rates
  4. Further splits on Fare and SibSp

This mirrors exactly what we expected from the exploratory analysis: gender is the most important predictor, followed by ticket class and age.

8 Model Evaluation — The Confusion Matrix

8.1 What is a Confusion Matrix?

After building a model, we compare its predictions against the actual values on the test set. The results are displayed in a confusion matrix.

For a binary classification problem (survived = 1, not survived = 0):

Actual: 0 Actual: 1
Predicted: 0 True Negative (TN) False Negative (FN)
Predicted: 1 False Positive (FP) True Positive (TP)

8.2 Titanic Example

          Actual
Predicted  0   1
        0  50  19   ← Predicted "not survived"
        1   4  17   ← Predicted "survived"
  • TN = 50: Correctly predicted “did not survive”
  • FN = 19: Person survived, but model predicted they didn’t (missed survivors)
  • FP = 4: Person didn’t survive, but model predicted they did
  • TP = 17: Correctly predicted “survived”

8.3 Accuracy

\[\text{Accuracy} = \frac{TP + TN}{TP + TN + FP + FN} = \frac{17 + 50}{90} \approx 74\%\]

Note to self: Accuracy is not always the best metric — it can be misleading when classes are imbalanced. For example, if only 5% of passengers survived, a model that always predicts “not survived” would be 95% accurate but completely useless. More metrics (precision, recall, F1) will be covered in future classes.

9 Key Takeaways from Class 3

  1. Data exploration is not optional — always understand your data before modelling.
  2. Plot both counts and proportions when comparing a variable to the target.
  3. Create derived variables when a threshold or transformation captures the signal better than the raw value (e.g. is_child from Age).
  4. Check relationships between predictor variables — not just between each predictor and the target. Correlated predictors carry redundant information.
  5. Always split into train and test before building a model. Never evaluate a model on data it was trained on.
  6. Set a random seed for reproducibility.
  7. Decision trees are interpretable — use them as an exploratory tool, not just a final model.
  8. Overfitting is when a model memorises the training data instead of learning general patterns.
