Lab 4a: Logistic Regression in R

Predicting Survival on the Titanic

1 What is this lab about?

In this lab we learn a new machine learning method called Logistic Regression.

Think of it like this:

Imagine you are trying to guess if it will rain today. You look at clues — dark clouds? windy? cold? You combine all the clues and come up with a score between 0 and 1. If the score is above 0.5, you say “yes, it will rain.” If it is below 0.5, you say “no, it will not rain.”

Logistic regression does exactly this — it looks at clues (columns in the data) and gives a probability (a number from 0 to 1) of something happening. Here, the “something” is: did this Titanic passenger survive?

2 Key Topics

  • What logistic regression is and how it is different from decision trees
  • Cleaning data before building a model (fixing empty values)
  • Building a logistic regression model with glm()
  • Converting a probability into a yes/no prediction using a threshold
  • Measuring model quality: accuracy, precision, recall
  • Understanding the ROC curve and AUC
  • What happens when you change the threshold
  • Why missing values (NA) are a problem for logistic regression

3 Step 1 — Load packages and data

library(titanic)      # the Titanic dataset
library(tidymodels)   # tools for model evaluation (roc_auc, roc_curve, etc.)

data(titanic_train)
head(titanic_train)
Note

tidymodels is a collection of tools that makes it easier to build, evaluate and compare machine learning models in R. We use it here mainly for the ROC curve and AUC score.

4 Step 2 — Clean the data

table(titanic_train$Embarked)

This counts how many passengers have each value in the Embarked column. Embarked means “which port did they board the ship from?” The values should be C, Q, or S.

titanic_train[titanic_train$Embarked == "",]$Embarked = "S"
Important

Why are we doing this?

Some passengers have an empty string "" for Embarked — meaning the data is missing. Logistic regression does not like missing or unexpected values. We replace them with "S" (Southampton) because that was the most common port — it is a reasonable guess.

🧒 It is like a form where someone left a field blank. We fill it in with the most common answer so the form is complete.

5 Step 3 — Split the data

set.seed(42)
nobs  <- nrow(titanic_train)
train <- sample(nobs, 0.9 * nobs)
test  <- setdiff(seq_len(nobs), train)

Same as Lab 3 — 90% of rows go to training (the model learns from these), 10% go to testing (we check how well the model works on new data). set.seed(42) makes sure we always get the same split.

6 Step 4 — Choose which columns to use

target       <- "Survived"
colsToIgnore <- c("PassengerId", "Name", "Cabin", "Ticket", "Age")
input        <- setdiff(names(titanic_train), c(target, colsToIgnore))
input
Warning

Why is Age excluded here — but not in Lab 3?

Logistic regression cannot handle missing values (NA) at all. If even one passenger in the test set has a missing age, the entire predict() call will return NA for that row.

In Lab 3 (decision trees) we fixed this by imputing (filling in) the missing ages. Here, the simpler fix is to just leave Age out of the model entirely. Later in the lab we test what happens when we include it — and we see the problem appear.

🧒 Imagine you are filling out a quiz but some answers are invisible. You cannot add up the score if some answers are missing. Logistic regression has the same problem.

Columns we keep as inputs: Pclass, Sex, SibSp, Parch, Fare, Embarked

train_data <- titanic_train[train, c(input, target)]
test_data  <- titanic_train[test,  c(input, target)]

We create two separate data frames — one for training and one for testing. This is cleaner than always writing titanic_train[train, ...] each time.

7 Step 5 — Build the logistic regression model

logistic_model <- glm(formula = Survived ~ .,
                      family  = binomial(link = "logit"),
                      data    = train_data)
summary(logistic_model)
logistic_model$coefficients
Note

Breaking this down piece by piece:

Part Meaning
glm() “Generalised Linear Model” — the R function for logistic regression
Survived ~ . “Predict Survived using all other columns”
family = binomial(link = "logit") “This is a yes/no prediction problem”
data = train_data “Learn from the training data only”

What are coefficients?

🧒 Think of coefficients as weights or importance scores for each clue. A large positive number means “this clue strongly suggests the person survived.” A large negative number means “this clue strongly suggests they did not survive.” A number close to zero means “this clue doesn’t help much.”

For example, you will likely see that Sexmale has a large negative coefficient — being male strongly reduced the chance of survival on the Titanic.

8 Step 6 — Make predictions

8.1 Get raw probabilities

pr <- predict(logistic_model, newdata = test_data, type = "response")
pr

type = "response" tells R to give us probabilities (numbers between 0 and 1), not raw scores.

🧒 Each number is like saying: “I am 73% sure this person survived” or “I am 12% sure this person survived.”

8.2 Convert probabilities to yes/no using a threshold

pr <- ifelse(pr > 0.5, 1, 0)

ifelse() says: if the probability is above 0.5, say 1 (survived). Otherwise, say 0 (did not survive).

🧒 It is like saying: “If I am more than 50% sure it will rain, I will bring an umbrella. Otherwise I won’t.”

The 0.5 here is called the threshold — we will change it later and see what happens.

9 Step 7 — Evaluate the model

9.1 Confusion matrix and accuracy

actuals <- factor(test_data$Survived, levels = c(1, 0))
pr      <- factor(pr, levels = c(1, 0))

tab1 <- table(pr, actuals)
tab1

accuracy <- 100 * sum(diag(tab1) / sum(tab1))
accuracy

We wrap the values in factor(..., levels = c(1, 0)) to tell R that 1 (survived) is the positive class — the thing we most care about predicting correctly.

Note

Confusion matrix reminder:

Actual: Survived (1) Actual: Died (0)
Predicted: Survived (1) ✅ True Positive (TP) ❌ False Positive (FP)
Predicted: Died (0) ❌ False Negative (FN) ✅ True Negative (TN)
  • Accuracy = how often we were right overall
  • But accuracy alone can be misleading! (More on this below)

10 Step 8 — ROC Curve and AUC

pr <- predict(logistic_model, newdata = test_data, type = "response")
test_data$predicted <- pr
test_data$Survived  <- factor(test_data$Survived, levels = c(1, 0))

auc <- roc_auc(test_data, Survived, predicted)
auc

roc <- roc_curve(test_data, Survived, predicted)
autoplot(roc)
Note

What is a ROC curve?

🧒 Imagine your model is a security guard at the door of a “survivors only” club. If the guard is very strict (high threshold), they only let in people they are almost 100% sure survived — but they will turn away some real survivors by mistake. If the guard is very relaxed (low threshold), they let almost everyone in — but some non-survivors sneak in too.

The ROC curve draws a picture of this trade-off for every possible threshold at once.

What is AUC?

AUC = “Area Under the Curve”. It is a single number that summarises the whole ROC curve.

AUC Value Meaning
1.0 Perfect model — always right
0.5 Useless model — no better than random guessing
0.7 – 0.8 Decent model
0.8 – 0.9 Good model
> 0.9 Excellent model

10.1 Precision-Recall curve

pr_c <- pr_curve(test_data, Survived, predicted)
autoplot(pr_c)
Note

Precision vs Recall — what is the difference?

These two measures tell you about different types of mistakes.

🧒 Imagine a doctor who tests patients for a disease:

  • Precision = “Of all the patients I said had the disease, how many actually had it?” (How accurate are my positive predictions?)
  • Recall = “Of all the patients who actually had the disease, how many did I catch?” (Did I miss anyone sick?)

In formulas:

\[ \text{Precision} = \frac{TP}{TP + FP} \]

\[ \text{Recall} = \frac{TP}{TP + FN} \]

There is always a trade-off between the two — which brings us to the next section.

11 Step 9 — Changing the threshold

This is one of the most important parts of the lab. We run the same model three times with different thresholds and compare the results.

11.1 What is the threshold?

pr <- ifelse(pr >= threshold, 1, 0)

The threshold is the cut-off point. If the model says the probability is above the threshold, we predict “survived”. Below it, we predict “died”.

🧒 Imagine you are sorting packages at a post office. If a package feels heavy enough, you mark it “fragile — handle with care.” The weight you choose as “heavy enough” is the threshold. Change that weight and different packages get marked.

11.2 Threshold = 0.5 (default)

threshold <- 0.5

Balanced — you need to be at least 50% sure before predicting survival.

11.3 Threshold = 0.1 (very low)

threshold <- 0.1

🧒 You only need to be 10% sure someone survived to predict they did. This means you will predict “survived” for almost everyone.

  • ✅ High recall — you will catch nearly all real survivors
  • ❌ Low precision — you will also wrongly predict survival for many people who did not survive

11.4 Threshold = 0.9 (very high)

threshold <- 0.9

🧒 You need to be 90% sure before predicting survival. This means you will only say “survived” for passengers you are very confident about.

  • ✅ High precision — when you say “survived”, you are almost always right
  • ❌ Low recall — you will miss many actual survivors
Important

The big trade-off:

Lowering the threshold → catches more survivors, but makes more false alarms
Raising the threshold → fewer false alarms, but misses more survivors

Which is better depends on the problem. In medical diagnosis, it is usually worse to miss a sick patient (low recall) than to over-diagnose (lower precision). In spam filters, it may be worse to block real emails (lower precision) than to let some spam through.

12 Step 10 — What happens when we include Age?

colsToIgnore <- c("PassengerId", "Name", "Cabin", "Ticket")  # Age NOT ignored
input <- setdiff(names(titanic_train), c(target, colsToIgnore))

When Age is included, the model will have NA (missing) values in the data. Logistic regression will silently drop those rows during training, and may produce NA predictions during testing.

The lab shows this on purpose so you can see the problem, then fixes it by putting Age back in colsToIgnore.

13 Step 11 — Removing weak predictors

summary(logistic_model)

Look at the Pr(>|z|) column in the summary — these are p-values.

Note

What is a p-value here?

🧒 A p-value tells you: “How likely is it that this column is just noise — that it has no real connection to survival?”

  • A small p-value (e.g. 0.001) means: “This column really matters.”
  • A large p-value (e.g. 0.8) means: “This column is probably not helping — it might just be random noise.”

Stars in the output (***, **, *) mean the column is statistically significant. No stars = probably not useful.

Columns with large p-values (Embarked, Parch, Fare) are removed:

input <- setdiff(input, c("Embarked", "Parch", "Fare"))

logistic_model2 <- glm(formula = Survived ~ .,
                       family  = binomial(link = "logit"),
                       data    = train_data)
summary(logistic_model2)

This builds a simpler model using only the most important columns. Simpler models are often better because they do not overfit to noise.

14 Key Equations

Logistic regression output (probability):

\[ P(\text{survived}) = \frac{1}{1 + e^{-(\beta_0 + \beta_1 x_1 + \beta_2 x_2 + \dots)}} \]

The \(\beta\) values are the coefficients from logistic_model$coefficients. The result is always between 0 and 1 — a probability.

Precision and Recall:

\[ \text{Precision} = \frac{TP}{TP + FP} \qquad \text{Recall} = \frac{TP}{TP + FN} \]

Accuracy:

\[ \text{Accuracy} = \frac{TP + TN}{TP + TN + FP + FN} \times 100 \]

15 Summary Points

  • Logistic regression predicts a probability (0 to 1) that something is true. It is good for yes/no questions.
  • The threshold (default 0.5) converts a probability into a yes/no answer. You can change it depending on whether you care more about precision or recall.
  • Logistic regression cannot handle missing values (NA) — either remove the column or impute the missing values before using it.
  • Always clean your data first — fix empty strings, missing values, and remove identifier columns.
  • Use summary() on your model to see which columns are actually useful. Columns with large p-values (no stars) can be safely removed.
  • AUC summarises the ROC curve in one number — closer to 1.0 is better.
  • Precision = “when I say yes, am I right?” / Recall = “did I catch all the real yeses?”
  • There is always a trade-off between precision and recall — changing the threshold moves you along this trade-off.

16 Difference from Lab 3 (Decision Trees)

Decision Tree (Lab 3) Logistic Regression (Lab 4a)
Output A category directly A probability, then converted
Missing values Can sometimes handle them Cannot — must remove or impute
Interpretability Easy to visualise as a tree Need to read coefficients
Threshold Not applicable You choose the cut-off

```r
summary(logistic_model)
```

Look at the `Pr(>|z|)` column in the summary — these are **p-values**.

::: {.callout-note}
**What is a p-value here?**

> 🧒 A p-value tells you: "How likely is it that this column is just
> noise — that it has no real connection to survival?"
>
> - A **small p-value** (e.g. `0.001`) means: "This column really matters."
> - A **large p-value** (e.g. `0.8`) means: "This column is probably
>   not helping — it might just be random noise."

Stars in the output (`***`, `**`, `*`) mean the column is statistically
significant. No stars = probably not useful.
:::

Columns with large p-values (`Embarked`, `Parch`, `Fare`) are removed:

```r
input <- setdiff(input, c("Embarked", "Parch", "Fare"))

logistic_model2 <- glm(formula = Survived ~ .,
                       family  = binomial(link = "logit"),
                       data    = train_data)
summary(logistic_model2)
```

This builds a **simpler model** using only the most important columns.
Simpler models are often better because they do not overfit to noise.

# Key Equations

**Logistic regression output (probability):**

$$
P(\text{survived}) = \frac{1}{1 + e^{-(\beta_0 + \beta_1 x_1 + \beta_2 x_2 + \dots)}}
$$

The $\beta$ values are the **coefficients** from `logistic_model$coefficients`.
The result is always between 0 and 1 — a probability.

**Precision and Recall:**

$$
\text{Precision} = \frac{TP}{TP + FP} \qquad \text{Recall} = \frac{TP}{TP + FN}
$$

**Accuracy:**

$$
\text{Accuracy} = \frac{TP + TN}{TP + TN + FP + FN} \times 100
$$

# Summary Points

- **Logistic regression** predicts a *probability* (0 to 1) that something
  is true. It is good for yes/no questions.
- The **threshold** (default 0.5) converts a probability into a yes/no answer.
  You can change it depending on whether you care more about precision or recall.
- **Logistic regression cannot handle missing values (`NA`)** — either remove
  the column or impute the missing values before using it.
- Always **clean your data first** — fix empty strings, missing values, and
  remove identifier columns.
- Use **`summary()`** on your model to see which columns are actually useful.
  Columns with large p-values (no stars) can be safely removed.
- **AUC** summarises the ROC curve in one number — closer to 1.0 is better.
- **Precision** = "when I say yes, am I right?" /
  **Recall** = "did I catch all the real yeses?"
- There is always a **trade-off between precision and recall** — changing
  the threshold moves you along this trade-off.
