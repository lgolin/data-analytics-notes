3 Communicating Your Results

3.1 Why Communication Matters

One of the most important skills for a data scientist is the ability to communicate results clearly to people who are not data scientists. The best model in the world has no value if nobody understands it or acts on it.

Key guidelines:

  • Focus on business value, not the machine learning technique. Nobody in business wants to hear about logistic regression – they want to know how many customers you will keep or how much money will be saved.
  • Know your audience. A process engineer understands box plots and statistics. A marketing analyst may not.
  • Many data science projects do not need machine learning. Creating good dashboards and visualisations is often more valuable.

3.2 Three Ways to Describe the Same Model

Here is an example of three ways to explain the same churn prediction model:

Summary Audience Quality
“Logistic regression with regularisation, recall 80%, precision 66%” Data scientist peers Too technical for business
“Identifies 80% of customers who will not return in 30 days; wrong 33% of the time” Marketing team Much better – explains the outcome
“Campaign will break even at 10% coupon redemption; at 20% redemption, incremental spend = €11,648, 1,500 extra customers retained” Business decision makers Best – connects model to money

Key idea: The third summary uses a mathematical model in Excel (not a machine learning model) to combine the data scientist’s metrics (recall, precision) with the marketing team’s assumptions (coupon cost, redemption rate). This allows both sides to be held accountable for their assumptions.

4 Clustering (Segmentation)

4.1 What Is Clustering?

Clustering (also called segmentation) is an unsupervised machine learning task. This means there is no target variable – we are not trying to predict anything. Instead, we are trying to find groups of data that are similar to each other.

Practical example: A supermarket might cluster its customers into groups like “young families”, “students”, and “pensioners”, each of which has different shopping habits. This allows the business to target each group differently.

⚠️ Important: Clusters will almost never be the same size. Expect one large cluster (the “average” customer) and several smaller, more interesting ones.

4.2 Why We Need to Standardise Data

Before clustering, we must standardise all numeric columns so they are on the same scale (0 to 1). Without this, columns with larger numbers (like square metres) will dominate over columns with smaller numbers (like number of bedrooms), even if the smaller column is more meaningful.

4.2.1 Standardisation Formula

\[x_{standardised} = \frac{x - \min(x)}{\max(x) - \min(x)}\]

This converts any numeric value to a number between 0 and 1.

In R:

standardise <- function(x) {
  (x - min(x)) / (max(x) - min(x))
}

iris$Sepal.Length <- standardise(iris$Sepal.Length)
iris$Sepal.Width  <- standardise(iris$Sepal.Width)
iris$Petal.Length <- standardise(iris$Petal.Length)
iris$Petal.Width  <- standardise(iris$Petal.Width)

After standardisation, the minimum of every column will be 0 and the maximum will be 1.

4.3 K-Means Clustering

4.3.1 How It Works

  1. You choose k (the number of clusters) before running the algorithm.
  2. The algorithm randomly places k centre points (called centroids).
  3. Each data point is assigned to the nearest centroid.
  4. Each centroid is moved to the average position of all points assigned to it.
  5. Steps 3 and 4 repeat until the assignments stop changing.

The algorithm is not guaranteed to find the perfect solution, so it is usually run multiple times (e.g. nstart = 10) and the best result is kept.

K-means tries to minimise the within-cluster variation – it wants the points inside each cluster to be as close together as possible.

4.3.2 R Code: K-Means on the Iris Dataset

# Run k-means with 3 clusters, 10 random starts
k_clusters <- kmeans(iris[, 1:4], centers = 3, nstart = 10)

# View results
k_clusters

The output shows: - The cluster means (centre of each cluster for each variable) - Which cluster each data point belongs to - The within-cluster sum of squares (lower = tighter clusters)

4.3.3 How to Choose k: The Elbow Method

If you do not know the number of clusters in advance, use the elbow method:

  1. Run k-means for k = 1, 2, 3, …, 10.
  2. Plot the total within-cluster sum of squares for each k.
  3. Find the point where the line “bends” like an elbow – that is your best k.
tot.withinss <- c()

for (k in 1:10) {
  k_clusters <- kmeans(iris[, 1:4], k, nstart = 10)
  tot.withinss <- c(tot.withinss, k_clusters$tot.withinss)
}

plot(main = "Selecting k: Elbow Method", tot.withinss, type = "l")

The curve drops steeply at first, then flattens out. The “elbow” (where the flattening begins) indicates the best number of clusters. For the iris dataset, the elbow is at k = 3.

4.4 Hierarchical Clustering

4.4.1 How It Works

Hierarchical clustering is a bottom-up approach:

  1. Every data point starts as its own cluster (150 points = 150 clusters).
  2. The two closest clusters are merged into one.
  3. Step 2 repeats until all points are in a single cluster.

The result is shown as a dendrogram (a tree diagram). You can then “cut” the tree at any height to get the number of clusters you want.

Advantage over k-means: You do not need to choose k in advance.

4.4.2 Reading a Dendrogram

  • Draw a horizontal line across the tree at any height.
  • Count how many vertical lines it crosses – that is the number of clusters.
  • The vertical distance between merges tells you how different the clusters are. Larger gaps = more distinct clusters.

4.4.3 R Code: Hierarchical Clustering on the Iris Dataset

# Calculate distances between points, then cluster
h_clusters <- hclust(dist(iris[, 1:4]))

# Cut the tree to get 3 clusters
clusterId <- cutree(h_clusters, 3)

# Add the cluster ID back to the data
iris$clusterId <- as.factor(clusterId)

# Plot the dendrogram
plot(h_clusters)

4.5 Interpreting Clusters

After clustering, a common question is: what do the clusters mean?

The answer is to use the same exploratory data analysis techniques from earlier weeks – but instead of colouring plots by a target variable, colour them by cluster ID.

library(ggplot2)

# Scatter plots coloured by cluster
p1 <- qplot(Sepal.Width, Sepal.Length, data = iris,
            colour = clusterId, main = "Sepal.Width vs Sepal.Length")

p2 <- qplot(Petal.Length, Petal.Width, data = iris,
            colour = clusterId, main = "Petal.Length vs Petal.Width")

gridExtra::grid.arrange(p1, p2)

Look for patterns in the plots: - Do certain clusters have higher values on certain variables? - Can you give a meaningful name to each cluster? (e.g. “Large flowers”, “Small flowers”)

4.5.1 Comparing Clusters to a Known Variable

Even though clustering does not use the Species column, we can check how well the clusters match it afterwards:

table(iris$clusterId, iris$Species)

A good match between clusters and the known species confirms the clustering is working well.

🔑 Key reminder: Clustering does not aim to create equal-sized groups. In real retail data, you might find 80% of customers in one large “average” cluster, and small but very interesting clusters of students, pensioners, or high-spending customers.

5 Recommendation Engines

5.1 What Are They?

Recommendation engines are systems that suggest products to customers based on what other customers bought. Examples:

  • “People who bought this also bought…”
  • “These items are commonly purchased together”

The most common algorithm for this is the Apriori algorithm, which finds patterns of items bought together in transactions (called market basket analysis).

5.2 Three Key Metrics

5.2.1 1. Support

How common is this item or item set?

\[\text{Support}(X) = \frac{\text{Number of transactions containing } X}{\text{Total number of transactions}}\]

Support is used to filter out rare items. If an item only appears in 0.01% of transactions, any rule involving it will not be useful in practice.

5.2.2 2. Confidence

If a customer buys X, how likely are they to also buy Y?

\[\text{Confidence}(X \Rightarrow Y) = \frac{\text{Support}(X \text{ and } Y)}{\text{Support}(X)}\]

This measures the strength of the rule “if X then Y”.

5.2.3 3. Lift

Is the relationship between X and Y stronger than what we would expect by chance?

\[\text{Lift}(X \Rightarrow Y) = \frac{\text{Support}(X \text{ and } Y)}{\text{Support}(X) \times \text{Support}(Y)}\]

Lift value Meaning
> 1 Buying X makes buying Y more likely than normal
= 1 X and Y are bought independently of each other
< 1 Buying X makes buying Y less likely than normal

🔑 Lift is the most important metric. Select rules with the highest lift. A lift of 3.0 means a customer is 3× more likely to buy Y when they have X in their basket.

5.3 R Code: Apriori on the Groceries Dataset

library(arules)
library(arulesViz)

# Load the built-in grocery dataset
data("Groceries")
summary(Groceries)

# Look at the first 3 transactions
inspect(head(Groceries, 3))

# See the most frequently bought items
freq <- itemFrequency(Groceries, type = "absolute")
freq <- sort(freq, decreasing = TRUE)
head(freq)

# Plot item frequencies for top 20 items
itemFrequencyPlot(Groceries, topN = 20)

5.3.1 Running the Apriori Algorithm

# Items must appear in at least 1% of transactions
# Rules must have at least 30% confidence
support    <- 0.01
confidence <- 0.30
parameters <- list(support = support, confidence = confidence, target = "rules")

rules <- apriori(Groceries, parameter = parameters)

# View rules as a data frame
dfRules <- data.frame(item_set = labels(rules), quality = rules@quality)
head(dfRules)

5.3.2 Plotting the Rules

# Scatter plot of all rules
plot(rules)

# Network graph of only the strongest rules (lift >= 3)
rulesLift <- subset(rules, lift >= 3.0)
plot(rulesLift, method = "graph")

5.4 Verifying Calculations Manually

You can verify what the algorithm is doing by calculating the metrics by hand:

# Convert sparse matrix to full data frame
dfTrans <- data.frame(as(Groceries, "matrix"))

# Calculate support, confidence, and lift for beef => root vegetables
support    <- sum(dfTrans$beef & dfTrans$root.vegetables) / nrow(dfTrans)
confidence <- sum(dfTrans$beef & dfTrans$root.vegetables) / sum(dfTrans$beef)
lift       <- support / ((sum(dfTrans$beef) / nrow(dfTrans)) *
                         (sum(dfTrans$root.vegetables) / nrow(dfTrans)))

5.5 Applying Recommendation Rules in Practice

Things to consider when using recommendations in a real business:

  • Transactional vs. member-based: You can group items by transaction ID (what is in one basket right now) or by member ID (what a person has bought over their whole history). Member-based rules can capture purchases that happen months apart (e.g. a cyclist buys a bike, then later a helmet, then cycling shoes).

  • Use segmentation + recommendations together: Run the recommendation engine separately on each customer segment. This gives much more targeted and relevant suggestions.

  • Substitute vs. complement products:

    • A substitute is a similar product (e.g. a different brand of TV). Goal: upsell (buy a better version).
    • A complement is a product bought alongside the main product (e.g. a TV cable). Goal: cross-sell (buy something extra).

  • Avoid bombarding customers. If someone just bought a TV, showing them more TV ads for 6 months is annoying and ineffective.

  • Ethics matter. A famous example: Target (US supermarket) identified pregnant women from their shopping habits and sent them baby product offers. One teenager’s father received the offers before he knew about the pregnancy. Always consider how customers will feel about personalised targeting.

6 Hyperparameter Optimisation

6.1 What Is a Hyperparameter?

In statistics, a parameter usually means a variable in your data (e.g. age, income).

In machine learning, a hyperparameter is a setting of the algorithm itself – something you choose before training, not learned from the data.

Examples:

Algorithm Hyperparameter examples
Decision Tree Splitting criterion (Gini, entropy)
Support Vector Machine Kernel function (linear, RBF)
Neural Network Number of layers, nodes per layer, learning rate
Random Forest Number of trees (ntree), features per tree (mtry)

The process of finding the best set of hyperparameters is called hyperparameter optimisation.

6.2 Methods for Hyperparameter Optimisation

6.2.1 Grid Search

Try every possible combination of hyperparameters. If you have 4 hyperparameters with 2, 5, 3, and 2 possible values each:

\[2 \times 5 \times 3 \times 2 = 60 \text{ experiments}\]

Grid search is exhaustive but very slow for large numbers of hyperparameters.

6.2.2 Random Search

Select combinations at random. Useful when you cannot afford to run all combinations. Less thorough than grid search, but much faster.

6.2.3 Bayesian Search

Starts like random search, then focuses on the most promising area based on the results seen so far. Think of it like the hot/cold game – it “gets warmer” and homes in on the best region.

6.3 R Code: Manual Hyperparameter Search (Random Forest)

library(randomForest)

# Define hyperparameter values to try
trees <- c(50, 200, 500)
mtrys <- c(4, 6)

best_accuracy <- 0

for (v1 in trees) {
  for (v2 in mtrys) {
    rf_model <- randomForest(as.factor(Survived) ~ .,
                             data  = train_data,
                             ntree = v1,
                             mtry  = v2)

    pr  <- predict(rf_model, newdata = test_data, type = "class")
    pr  <- factor(pr, levels = c(1, 0))
    tab <- table(pr, actuals)
    accuracy <- 100 * sum(diag(tab)) / sum(tab)

    if (accuracy > best_accuracy) {
      best_tree     <- v1
      best_mtry     <- v2
      best_accuracy <- accuracy
    }
  }
}

print(paste0("Best accuracy = ", round(best_accuracy, 2),
             ", ntree = ", best_tree,
             ", mtry = ", best_mtry))

This runs 6 experiments (3 tree values × 2 mtry values). In practice, use the caret or tidymodels packages to do this automatically and more rigorously.

6.4 Important Warnings

6.4.1 You Need THREE Data Splits for Hyperparameter Optimisation

When optimising hyperparameters, do not keep evaluating on the same test set – you are effectively peeking at the answers. Use:

  1. Training set – train the model
  2. Validation set – compare hyperparameter combinations
  3. Test set – used only once at the very end to report final accuracy

A typical split might be 70% / 15% / 15%.

6.4.2 Hyperparameter Optimisation Is NOT the First Step

⚠️ Feature engineering and data quality matter much more than hyperparameter tuning. Most algorithms give reasonable results with default settings. Only optimise hyperparameters once your data preparation is solid.

6.5 AutoML

Some tools (e.g. h2o.ai, AWS SageMaker) automate the whole process: selecting the algorithm AND tuning the hyperparameters. This is called AutoML.

  • Useful for extracting the last small improvement in performance.
  • Can run for days on large datasets using multiple cloud machines.
  • The improvement is usually small (often less than 1% accuracy difference between models).
  • Should be the last step in your workflow, not the first.

7 Key Formulas Summary

Formula Description
\(x_{std} = \frac{x - \min(x)}{\max(x) - \min(x)}\) Standardisation (min-max scaling)
\(\text{Support}(X) = \frac{\text{count}(X)}{N}\) Fraction of transactions containing X
\(\text{Confidence}(X \Rightarrow Y) = \frac{\text{Support}(X, Y)}{\text{Support}(X)}\) Probability of Y given X is in the basket
\(\text{Lift}(X \Rightarrow Y) = \frac{\text{Support}(X, Y)}{\text{Support}(X) \times \text{Support}(Y)}\) How much more likely Y is when X is present

9 Summary Points

  • Communication is a core skill. Always connect your model to business outcomes (revenue, cost savings, customer retention).

  • Clustering is unsupervised – no target variable. It groups similar data points together. Always standardise data before clustering.

  • K-means requires you to choose k in advance. Use the elbow method to help. Run multiple times with nstart to avoid bad random starts.

  • Hierarchical clustering builds a tree (dendrogram) bottom-up, so you can choose the number of clusters after running it.

  • Clusters are not equal in size. The most interesting clusters are often the small ones.

  • After clustering, use exploratory data analysis (scatter plots, box plots) to understand what each cluster represents.

  • Recommendation engines use three metrics:

    • Support – how common is this item?
    • Confidence – given X, how likely is Y?
    • Lift – is the X→Y relationship stronger than expected by chance? (Want > 1)

  • Hyperparameters are settings of the algorithm (not the data). Finding the best ones is called hyperparameter optimisation.

  • Grid search tries all combinations. Random search tries random ones. Bayesian search focuses on promising areas.

  • Always split data into 3 parts (train / validation / test) when doing hyperparameter optimisation.

  • Feature engineering > hyperparameter tuning. Fix your data first.

## K-Means Clustering

### How It Works

1. You choose **k** (the number of clusters) before running the algorithm.
2. The algorithm randomly places **k centre points** (called centroids).
3. Each data point is assigned to the nearest centroid.
4. Each centroid is moved to the **average position** of all points assigned to it.
5. Steps 3 and 4 repeat until the assignments stop changing.

The algorithm is **not guaranteed** to find the perfect solution, so it is usually run multiple times (e.g. `nstart = 10`) and the best result is kept.

**K-means tries to minimise the within-cluster variation** – it wants the points inside each cluster to be as close together as possible.

### R Code: K-Means on the Iris Dataset

```r
# Run k-means with 3 clusters, 10 random starts
k_clusters <- kmeans(iris[, 1:4], centers = 3, nstart = 10)

# View results
k_clusters
```

The output shows:
- The **cluster means** (centre of each cluster for each variable)
- Which cluster each data point belongs to
- The **within-cluster sum of squares** (lower = tighter clusters)

### How to Choose k: The Elbow Method

If you do not know the number of clusters in advance, use the **elbow method**:

1. Run k-means for k = 1, 2, 3, ..., 10.
2. Plot the **total within-cluster sum of squares** for each k.
3. Find the point where the line "bends" like an elbow – that is your best k.

```r
tot.withinss <- c()

for (k in 1:10) {
  k_clusters <- kmeans(iris[, 1:4], k, nstart = 10)
  tot.withinss <- c(tot.withinss, k_clusters$tot.withinss)
}

plot(main = "Selecting k: Elbow Method", tot.withinss, type = "l")
```

> The curve drops steeply at first, then flattens out. The "elbow" (where the flattening begins) indicates the best number of clusters. For the iris dataset, the elbow is at **k = 3**.

## Hierarchical Clustering

### How It Works

Hierarchical clustering is a **bottom-up** approach:

1. Every data point starts as its own cluster (150 points = 150 clusters).
2. The two **closest** clusters are merged into one.
3. Step 2 repeats until all points are in a single cluster.

The result is shown as a **dendrogram** (a tree diagram). You can then "cut" the tree at any height to get the number of clusters you want.

**Advantage over k-means:** You do **not** need to choose k in advance.

### Reading a Dendrogram

- Draw a **horizontal line** across the tree at any height.
- Count how many vertical lines it crosses – that is the number of clusters.
- The **vertical distance** between merges tells you how different the clusters are. Larger gaps = more distinct clusters.

### R Code: Hierarchical Clustering on the Iris Dataset

```r
# Calculate distances between points, then cluster
h_clusters <- hclust(dist(iris[, 1:4]))

# Cut the tree to get 3 clusters
clusterId <- cutree(h_clusters, 3)

# Add the cluster ID back to the data
iris$clusterId <- as.factor(clusterId)

# Plot the dendrogram
plot(h_clusters)
```

## Interpreting Clusters

After clustering, a common question is: **what do the clusters mean?**

# Recommendation Engines

## What Are They?

Recommendation engines are systems that suggest products to customers based on what other customers bought. Examples:

- "People who bought this also bought…"
- "These items are commonly purchased together"

The most common algorithm for this is the **Apriori algorithm**, which finds patterns of items bought together in transactions (called **market basket analysis**).

## Three Key Metrics

### 1. Support

**How common is this item or item set?**

$$\text{Support}(X) = \frac{\text{Number of transactions containing } X}{\text{Total number of transactions}}$$

Support is used to **filter out rare items**. If an item only appears in 0.01% of transactions, any rule involving it will not be useful in practice.

### 2. Confidence

**If a customer buys X, how likely are they to also buy Y?**

$$\text{Confidence}(X \Rightarrow Y) = \frac{\text{Support}(X \text{ and } Y)}{\text{Support}(X)}$$

This measures the **strength of the rule** "if X then Y".

### 3. Lift

**Is the relationship between X and Y stronger than what we would expect by chance?**

$$\text{Lift}(X \Rightarrow Y) = \frac{\text{Support}(X \text{ and } Y)}{\text{Support}(X) \times \text{Support}(Y)}$$

| Lift value | Meaning |
|------------|---------|
| > 1 | Buying X makes buying Y **more likely** than normal |
| = 1 | X and Y are bought **independently** of each other |
| < 1 | Buying X makes buying Y **less likely** than normal |

> 🔑 **Lift is the most important metric.** Select rules with the highest lift. A lift of 3.0 means a customer is 3× more likely to buy Y when they have X in their basket.

## Verifying Calculations Manually

You can verify what the algorithm is doing by calculating the metrics by hand:

```r
# Convert sparse matrix to full data frame
dfTrans <- data.frame(as(Groceries, "matrix"))

# Calculate support, confidence, and lift for beef => root vegetables
support    <- sum(dfTrans$beef & dfTrans$root.vegetables) / nrow(dfTrans)
confidence <- sum(dfTrans$beef & dfTrans$root.vegetables) / sum(dfTrans$beef)
lift       <- support / ((sum(dfTrans$beef) / nrow(dfTrans)) *
                         (sum(dfTrans$root.vegetables) / nrow(dfTrans)))
```

## Applying Recommendation Rules in Practice

Things to consider when using recommendations in a real business:

- **Transactional vs. member-based:** You can group items by transaction ID (what is in one basket right now) or by member ID (what a person has bought over their whole history). Member-based rules can capture purchases that happen months apart (e.g. a cyclist buys a bike, then later a helmet, then cycling shoes).

- **Use segmentation + recommendations together:** Run the recommendation engine **separately on each customer segment**. This gives much more targeted and relevant suggestions.

- **Substitute vs. complement products:**
  - A **substitute** is a similar product (e.g. a different brand of TV). Goal: **upsell** (buy a better version).
  - A **complement** is a product bought alongside the main product (e.g. a TV cable). Goal: **cross-sell** (buy something extra).

- **Avoid bombarding customers.** If someone just bought a TV, showing them more TV ads for 6 months is annoying and ineffective.

- **Ethics matter.** A famous example: Target (US supermarket) identified pregnant women from their shopping habits and sent them baby product offers. One teenager's father received the offers before he knew about the pregnancy. Always consider how customers will feel about personalised targeting.

# Hyperparameter Optimisation

## What Is a Hyperparameter?

In statistics, a **parameter** usually means a variable in your data (e.g. `age`, `income`).

In machine learning, a **hyperparameter** is a **setting of the algorithm itself** – something you choose before training, not learned from the data.

**Examples:**

| Algorithm | Hyperparameter examples |
|-----------|------------------------|
| Decision Tree | Splitting criterion (Gini, entropy) |
| Support Vector Machine | Kernel function (linear, RBF) |
| Neural Network | Number of layers, nodes per layer, learning rate |
| Random Forest | Number of trees (`ntree`), features per tree (`mtry`) |

The process of finding the **best** set of hyperparameters is called **hyperparameter optimisation**.

## Methods for Hyperparameter Optimisation

### Grid Search

Try **every possible combination** of hyperparameters. If you have 4 hyperparameters with 2, 5, 3, and 2 possible values each:

$$2 \times 5 \times 3 \times 2 = 60 \text{ experiments}$$

Grid search is **exhaustive** but **very slow** for large numbers of hyperparameters.

### Random Search

Select combinations **at random**. Useful when you cannot afford to run all combinations. Less thorough than grid search, but much faster.

### Bayesian Search

Starts like random search, then **focuses on the most promising area** based on the results seen so far. Think of it like the hot/cold game – it "gets warmer" and homes in on the best region.

## R Code: Manual Hyperparameter Search (Random Forest)

```r
library(randomForest)

# Define hyperparameter values to try
trees <- c(50, 200, 500)
mtrys <- c(4, 6)

best_accuracy <- 0

for (v1 in trees) {
  for (v2 in mtrys) {
    rf_model <- randomForest(as.factor(Survived) ~ .,
                             data  = train_data,
                             ntree = v1,
                             mtry  = v2)

    pr  <- predict(rf_model, newdata = test_data, type = "class")
    pr  <- factor(pr, levels = c(1, 0))
    tab <- table(pr, actuals)
    accuracy <- 100 * sum(diag(tab)) / sum(tab)

    if (accuracy > best_accuracy) {
      best_tree     <- v1
      best_mtry     <- v2
      best_accuracy <- accuracy
    }
  }
}

print(paste0("Best accuracy = ", round(best_accuracy, 2),
             ", ntree = ", best_tree,
             ", mtry = ", best_mtry))
```

> This runs 6 experiments (3 tree values × 2 mtry values). In practice, use the `caret` or `tidymodels` packages to do this automatically and more rigorously.

## Important Warnings

### You Need THREE Data Splits for Hyperparameter Optimisation

When optimising hyperparameters, do **not** keep evaluating on the same test set – you are effectively peeking at the answers. Use:

1. **Training set** – train the model
2. **Validation set** – compare hyperparameter combinations
3. **Test set** – used **only once** at the very end to report final accuracy

A typical split might be 70% / 15% / 15%.

### Hyperparameter Optimisation Is NOT the First Step

> ⚠️ Feature engineering and data quality matter **much more** than hyperparameter tuning. Most algorithms give reasonable results with default settings. Only optimise hyperparameters once your data preparation is solid.

## AutoML

Some tools (e.g. **h2o.ai**, **AWS SageMaker**) automate the whole process: selecting the algorithm AND tuning the hyperparameters. This is called **AutoML**.

- Useful for extracting the last small improvement in performance.
- Can run for days on large datasets using multiple cloud machines.
- The improvement is usually small (often less than 1% accuracy difference between models).
- Should be the **last** step in your workflow, not the first.

|---------|-------------|
| $x_{std} = \frac{x - \min(x)}{\max(x) - \min(x)}$ | Standardisation (min-max scaling) |

# Summary Points

- **Communication is a core skill.** Always connect your model to business outcomes (revenue, cost savings, customer retention).

- **Clustering is unsupervised** – no target variable. It groups similar data points together. Always **standardise** data before clustering.

- **K-means** requires you to choose k in advance. Use the **elbow method** to help. Run multiple times with `nstart` to avoid bad random starts.

- **Hierarchical clustering** builds a tree (dendrogram) bottom-up, so you can choose the number of clusters after running it.

- **Clusters are not equal in size.** The most interesting clusters are often the small ones.

- **After clustering, use exploratory data analysis** (scatter plots, box plots) to understand what each cluster represents.

- **Recommendation engines use three metrics:**
  - **Support** – how common is this item?
  - **Confidence** – given X, how likely is Y?
  - **Lift** – is the X→Y relationship stronger than expected by chance? (Want > 1)

- **Hyperparameters** are settings of the algorithm (not the data). Finding the best ones is called **hyperparameter optimisation**.

- **Grid search** tries all combinations. **Random search** tries random ones. **Bayesian search** focuses on promising areas.

- Always split data into **3 parts** (train / validation / test) when doing hyperparameter optimisation.

- **Feature engineering > hyperparameter tuning.** Fix your data first.
