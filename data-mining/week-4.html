1 Part 1 — Why Accuracy Alone Is Not Enough

1.1 The Problem With Accuracy

Accuracy is the most obvious measure: the percentage of predictions that were correct.

\[\text{Accuracy} = \frac{\text{Correct Predictions}}{\text{Total Predictions}} = \frac{TP + TN}{TP + TN + FP + FN}\]

It is easy to understand — 100% means every prediction was right. But there are two situations where accuracy can fool you badly.

1.1.1 Situation 1 — Unequal Cost of Errors

In many real-world problems, the two types of errors are not equally bad.

NoteExample — Medical Diagnosis
  • A false positive (predicting “sick” when the person is healthy) is upsetting but not life-threatening — the patient might get further tests and eventually be cleared.
  • A false negative (predicting “healthy” when the person is actually sick) can be fatal, because treatment is delayed.

These two errors are not the same, but accuracy treats them as identical.

Important⭐ Professor’s real-world example — The Irish BreastCheck scandal

Professor’s note: “We had a scandal in the medical industry for our BreastCheck, which was an early screening programme to detect cancers… it probably managed to diagnose correctly whether a person was sick or healthy 95, 98, 99%. But it missed a number of people who had cancerous tumours and subsequently missed out on treatment, and by the time their cases progressed and they got the treatment, their cancers had spread, and the number of people unfortunately died from it.”

The professor’s point: “It doesn’t really matter… whether you’re 99% accurate or not. It’s the number of people that are diagnosed as healthy but in fact they’re sick. That’s the only metric that you’re worried about.”

This kind of story is exactly the type of thing the professor uses to build an exam question around — be ready to explain why recall matters more than accuracy in medical screening.

1.1.2 Situation 2 — Imbalanced Datasets

When one class is much more common than the other, a lazy model can score very high on accuracy while being useless.

WarningCommon Mistake

If 1% of transactions are fraudulent, a model that predicts “not fraud” for every transaction is 99% accurate — and catches zero fraud cases. Accuracy looks great; the model is worthless.

Professor’s note: “A machine learning model could just turn on and basically say, 100% everyone’s healthy, and I wouldn’t catch those records. So in a lot of cases, accuracy is actually a misleading evaluation metric to use.”

Real examples of imbalanced data the professor mentioned:

Domain Positive class (what we want to catch) Typical prevalence
Credit card fraud Fraudulent transactions < 0.1%
Medical screening (cancer) Person has disease ~5%
Spam filtering Spam emails Varies, often a minority
Manufacturing defects Defective items < 1%

2 Part 2 — The Confusion Matrix

A confusion matrix is a 2×2 table that breaks predictions into four categories. It is the foundation of every other classification metric.

2.1 The Four Outcomes

We always pick one class to call Positive (the thing we are trying to detect). The other is Negative.

Actual Positive (1) Actual Negative (0)
Predicted Positive (1) True Positive (TP) ✅ False Positive (FP) ❌
Predicted Negative (0) False Negative (FN) ❌ True Negative (TN) ✅
NoteHow to Remember “True” and “False”
  • Words that start with “True” → the model was correct.
  • Words that start with “False” → the model was wrong.
  • The second word (Positive/Negative) tells you what the model predicted.

So “False Negative” = the model predicted Negative, but was wrong (i.e. the actual answer was Positive).

2.2 The Titanic Example (Used Throughout Class)

The decision tree model from Week 3 produced this confusion matrix (with Survived = 1 as the Positive class, after re-ordering):

         actuals
pr         1    0
   1      17    4      ← Model predicted SURVIVED
   0      19   50      ← Model predicted DID NOT SURVIVE

Reading this:

Cell Count Meaning
TP 17 Did survive AND the model said they survived ✅
FP 4 Did not survive, but the model said they did ❌
FN 19 Did survive, but the model said they didn’t ❌
TN 50 Did not survive AND the model agreed ✅

Total accuracy = (17 + 50) / 90 = 74.4%

Important⭐ Exam-Critical Detail — The R Default vs. The “Standard” Layout

R by default puts the class labelled 0 in the first row/column. Most textbooks and Python libraries put the Positive class (1) first, because that is the class we care most about.

Professor’s note: “Unfortunately R kind of reverses your confusion matrix… so we’re going to reverse that, because we want to treat our first class, or class where it’s equal to 1, as a primary class for calculating other metrics.”

The R code to flip the matrix is:

actuals <- factor(test_data$Survived, levels=c(1,0))
pr      <- factor(pr, levels=c(1,0))
tab     <- table(pr, actuals)

If you are ever asked to calculate precision and recall from an R confusion matrix, check which class is the positive one first.

3 Part 3 — Precision, Recall, and F1

Three metrics to memorise. Each one answers a different question.

3.1 Precision — “When the model says YES, how often is it right?”

\[\text{Precision} = \frac{TP}{TP + FP}\]

Precision looks at the predicted column. Of all the cases the model flagged as Positive, what fraction actually were Positive?

Titanic example: Precision = 17 / (17 + 4) = 0.81

NotePlain-English Analogy — Precision is like a fishing net’s “keep rate”

If you cast a net and pull in 100 things, and 81 of them are fish (the rest are seaweed), your precision is 81%. The net is quite clean — when it catches something, it’s usually a fish.

3.2 Recall (Sensitivity) — “Of all the real YES cases, how many did the model find?”

\[\text{Recall} = \frac{TP}{TP + FN}\]

Recall looks at the actual column. Of all the cases that truly were Positive, what fraction did the model catch?

Titanic example: Recall = 17 / (17 + 19) = 0.47

NotePlain-English Analogy — Recall is like a search party’s “find rate”

If 100 people are lost in a forest and the search party brings back 47 of them, the recall is 47%. It doesn’t matter how many false alarms there were — only how many of the real lost people they found.

Important⭐ Mnemonic the Professor Gave for the Exam

Professor’s note: “Precision and recall are two things that a lot of people have a bit of confusion about, and it can be hard to remember the definitions. The way I try and remember recall is: recall just looks at the actual values, and precision looks at the predicted values.”

  • Recall → look at the row where the Actual = 1 (focus on actuals)
  • Precision → look at the row where the Predicted = 1 (focus on predictions)

Expect to calculate both by hand from a confusion matrix on the exam.

3.3 F1 Score — The Balanced Summary

\[F_1 = 2 \times \frac{\text{Precision} \times \text{Recall}}{\text{Precision} + \text{Recall}}\]

F1 is the harmonic mean of precision and recall. It gives a single number that only gets high when both precision and recall are high.

Titanic example: F1 = 2 × (0.81 × 0.47) / (0.81 + 0.47) ≈ 0.60

WarningWhy Harmonic Mean and Not Simple Average?

Because the harmonic mean punishes extreme imbalance. Consider a model with Precision = 1.0 and Recall = 0.0:

  • Simple average: 0.5 (looks okay)
  • F1 score: 0.0 (correctly shows the model is broken)

Professor’s note: “If precision is 0.99 but the recall is 0.01, it just doesn’t take an average, call it 50% and call it done. It basically will allocate a very low F1 score on that.”

3.4 Quick Reference Table

Metric Formula Answers the question When it matters most
Accuracy (TP+TN)/Total “Overall, how often right?” Balanced data, equal costs
Precision TP/(TP+FP) “When I say YES, am I right?” When false alarms are expensive
Recall TP/(TP+FN) “Did I catch all the real YES cases?” When missing a case is expensive
F1 Harmonic mean of P & R “Are both P and R good?” Imbalanced data, both errors matter

4 Part 4 — Communicating Metrics in Plain English

Important⭐ This Is One of the Professor’s Biggest Points — Highly Likely to Appear in the Exam

Professor’s note: “I can’t highlight this enough… very early on in my career, when I had finished my master’s in data science, I had all these tools and all these knowledge, and I wanted to demonstrate these, and I put things like F1 scores inside in presentations… and my presentation basically fell flat. Too many data scientists cannot converse with the people that are the decision makers, and if they want their models to get into production, they need to find a way to communicate what their model will actually do.”

The professor’s point: a model is only useful if the business understands what it does. Do not walk into a meeting talking about F1 scores.

4.1 How to Translate Each Metric for a Business Audience

Imagine the Titanic model is actually a disease-screening model (1 = sick, 0 = healthy). Its precision is 0.81 and recall is 0.47.

Technical statement Plain-English translation
“Recall is 0.47” “Of the people who are actually sick, the model finds 47% of them.”
“Precision is 0.81” “Of the people the model says are sick, 81% really are sick.”
“F1 is 0.60” (Don’t use F1 with business — use precision and recall separately.)

4.2 Is This a “Good” Model?

Depends entirely on context:

  • Serious disease (like cancer): missing sick patients is unacceptable → recall is king → this model is bad (only 47%).
  • Quick screening tool (e.g., COVID thermometer gun at the door): high precision with some missed cases may be acceptable → “better than nothing” says the professor.

5 Part 5 — ROC Curve and AUC

5.1 What Is the ROC Curve?

The Receiver Operating Characteristic (ROC) curve is a plot that shows how well a classifier can distinguish between the two classes across all possible thresholds.

NoteFun Historical Analogy

The ROC curve was invented for radar operators in World War II who had to decide whether a blip on the screen was a real enemy plane or just noise. That’s why it’s called “Receiver Operating Characteristic.”

  • Y-axis: Sensitivity = Recall = TP / (TP + FN)
  • X-axis: 1 − Specificity = FP / (FP + TN) = False Positive Rate

A perfect model would go straight up the left side and across the top — 100% recall with 0% false positives.

5.2 AUC — Area Under the Curve

AUC is a single number summarising the ROC curve.

AUC value Interpretation
1.0 Perfect classifier
0.9 – 1.0 Excellent
0.8 – 0.9 Good
0.7 – 0.8 Fair
0.5 No better than random guessing
< 0.5 Worse than random (predictions inverted)

Titanic decision tree AUC: 0.764 (decent but not great)

6 Part 6 — The Precision / Recall Trade-Off

6.1 The 0.5 Threshold Is Not Sacred

Most classifiers output a probability between 0 and 1. By default we convert this to a class using:

If prob >= 0.5 → predict Positive
Else → predict Negative

But 0.5 is just a convention. By moving the threshold, we can prioritise precision or recall.

NoteThreshold as a Volume Knob

Think of the threshold as a volume knob for how eager the model is to say “Positive”:

  • Low threshold (e.g., 0.1) — the model shouts “Positive!” at almost everything. Catches most real positives (high recall) but also raises lots of false alarms (low precision).
  • High threshold (e.g., 0.9) — the model only says “Positive” when 90%+ sure. Rarely wrong when it does (high precision) but misses many real cases (low recall).

6.2 Actual Results from Lab 4a (Logistic Regression on Titanic)

Threshold Accuracy Precision Recall Character
0.1 ~65% 0.54 0.92 “Paranoid” — catches almost everyone, lots of false alarms
0.5 ~73% 0.73 0.53 Balanced default
0.9 ~64% 1.00 0.11 “Cautious” — only when very sure

You cannot push up both precision AND recall — improving one almost always costs the other.

Important⭐ Exam Application

If the exam describes a scenario, you should be able to recommend a threshold:

Scenario Prioritise Move threshold…
Cancer screening — don’t miss a case Recall Lower
Spam filter — don’t send real email to junk Precision Higher
Fraud detection — catch as many as possible Recall Lower
Legal evidence — must be certain Precision Higher

7 Part 7 — Logistic Regression

7.1 What Is It?

Logistic regression is a classification algorithm (despite “regression” in the name). It takes the framework of linear regression but maps the output through a sigmoid (logistic) function so the result is a probability between 0 and 1.

Linear regression formula: \[z = \beta_0 + \beta_1 x_1 + \beta_2 x_2 + \dots + \beta_n x_n\]

Sigmoid transformation: \[P(y = 1) = \frac{1}{1 + e^{-z}}\]

The output is the probability that the observation belongs to the positive class.

7.2 Why It Is Popular

  • Interpretable coefficients — each \(\beta\) tells you how the variable affects the probability.
  • Fast to train, even on large data.
  • Statistical significance is reported via p-values.
  • A good baseline before trying fancy algorithms.

7.3 Key Points From Class

7.3.1 Interpreting the Coefficients

  • Negative coefficient → higher values of that feature decrease the probability of survival.
  • Positive coefficient → higher values increase the probability of survival.

Professor’s note: “If you look at the way the formula for logistic regression is … if the coefficient is negative, that means that a higher value is an indication that the person did not survive. So in this case, Pclass is negative, which means the higher value for Pclass, the less likely somebody survived.”

7.3.2 Handling Categorical Variables

When R sees a categorical variable like Sex:

  • It creates indicator variables (Sexmale = 0 or 1).
  • One category is absorbed into the intercept — typically the alphabetically first value.
  • So Sexfemale has no explicit coefficient — it’s wrapped into the intercept.
NoteWhy the Intercept Is So Large

The intercept effectively represents the “baseline person” — in the Titanic case, a female passenger who embarked at Cherbourg. Both conditions are associated with higher survival, so the intercept comes out large and positive.

7.3.3 The Missing-Data Problem

Warning⚠️ Common Mistake — Don’t Forget to Handle NA’s!

Logistic regression cannot handle missing values. If you train a model using a column with NAs (like Age in Titanic), the predict() function will return NA for those rows — and you’ll get a broken confusion matrix.

Two options:

  1. Drop the column entirely (quick fix)
  2. Impute the missing values (better: use the mean of the training set only, never the full dataset)

Professor’s note: “We only calculate the average of the age across the [training] split of the records… We don’t take the average over all of them, because that would mean that we’re leaking data into the [model].”

7.3.4 Occam’s Razor — Simpler Is Often Better

In the lab, removing Embarked, Parch, and Fare (all statistically insignificant) improved accuracy slightly — from 73% to 74%.

Why? These variables added noise without adding signal. Correlated features (like Fare and Pclass) don’t both need to be in the model.

Important⭐ The Concept of Multicollinearity

A variable may appear “insignificant” in the summary not because it has no relationship with the outcome, but because another variable in the model already captures the same information.

Example: Embarked looks insignificant because Pclass already explains most of its predictive power (people from Cherbourg were more likely to be first class).

8 Part 8 — Evaluation Metrics for Regression

For numerical targets (predicting a price, a temperature, a spend) we use different metrics.

8.1 The Four Regression Metrics

Metric Formula Units Key property
MAE \(\frac{1}{n}\sum \lvert y_i - \hat{y}_i \rvert\) Same as y Easy to explain, treats all errors equally
MSE \(\frac{1}{n}\sum (y_i - \hat{y}_i)^2\) y² Penalises big errors heavily
RMSE \(\sqrt{\text{MSE}}\) Same as y Penalises big errors, yet still interpretable
R² Correlation(y, ŷ)² Unitless 0–1 Proportion of variance explained

8.2 MAE — Mean Absolute Error

\[\text{MAE} = \frac{1}{n} \sum_{i=1}^{n} |y_i - \hat{y}_i|\]

NoteProfessor’s Example — Customer Spending Model

Professor’s note: “If we said that we had a model that was trying to predict how much a person was going to spend in our store in the next month, and if it basically said that the mean absolute error was 15, then the model is accurate plus or minus 15 euros.”

MAE’s big advantage: it is in the same units as the target. Easy to communicate to business.

8.3 MSE — Mean Squared Error

\[\text{MSE} = \frac{1}{n} \sum_{i=1}^{n} (y_i - \hat{y}_i)^2\]

Squaring has two effects:

  1. Removes negatives (a -5 error and a +5 error both contribute 25).
  2. Big errors are punished more than small ones (2² = 4 vs 5² = 25).

Good for models where you really don’t want huge misses.

8.4 RMSE — Root Mean Squared Error

\[\text{RMSE} = \sqrt{\text{MSE}}\]

Squaring puts the error back in the same units as y, but RMSE still penalises large errors more than MAE does. In practice, RMSE ≥ MAE always.

8.5 R² — Coefficient of Determination

Tells you what fraction of the variance in the target is explained by the model.

  • R² = 1 → perfect fit
  • R² = 0 → model is no better than just predicting the mean
  • R² < 0 → model is worse than just predicting the mean (yes, this can happen on test data!)

For most regression tasks, R² and RMSE are the preferred metrics.

9 Part 9 — The Bias–Variance Trade-Off

This is the single most important concept in the week. The professor came back to it many times.

9.1 The Two Evils

Concept What it means Training error Test error Also called
Bias Model too simple; misses the pattern High High Underfitting
Variance Model too complex; memorises noise Low High Overfitting
NoteAnalogy — The Student Studying for an Exam
  • High bias (underfit) — the student only skimmed the textbook. They don’t know the material. They do badly on the practice exam and the real exam.
  • Good fit — the student studied properly and understands the concepts. They do well on both practice and real exams.
  • High variance (overfit) — the student memorised every word of the practice exam answers. On the practice exam they score 100%. On the real exam (with slightly different questions) they score badly because they never learned the underlying concepts.

9.2 Visualising It (from Lab 4b)

Using synthetic data, three polynomial regression models were fitted:

Model Formula R² (training) MAE (training) Real-world behaviour
1 \(y = \beta_0 + \beta_1 x\) 0.17 3.45 Underfits — a straight line cannot capture the curve.
2 \(y = \beta_0 + \beta_1 x + \beta_2 x^2 + \beta_3 x^3\) 0.79 1.54 Good fit — captures the general shape.
3 Degree 6 polynomial 0.97 0.62 Overfits — hits every training point but oscillates wildly outside the training range.
Warning⚠️ The Deceptive “Best” Model

Model 3 has the best training metrics (R² = 0.97, lowest MAE). It looks like the winner. But it would fail catastrophically on new data, because it has memorised the random noise in the training set, not the underlying relationship.

This is the most common failure mode in applied machine learning. Never trust training metrics alone.

9.3 How to Detect Overfitting

Three practical tests:

  1. Compare training vs test performance. A big gap (great on training, mediocre on test) means the model overfit.
  2. Change the random seed and refit. If results vary wildly, the model is unstable.
  3. Use cross-validation (covered in a later class) for a systematic check.

10 Part 10 — Survey of Other ML Algorithms

The professor gave a high-level tour of four more algorithms. Each uses a different “philosophy” to make predictions. All were tested on the Titanic data.

Algorithm Core idea Titanic accuracy Strengths Weaknesses

10.2 Naive Bayes

10.2.1 The Big Idea

Start with a prior belief, then update it each time new evidence arrives.

\[\text{Posterior} = \frac{\text{Prior} \times \text{Likelihood}}{\text{Prior} \times \text{Likelihood} + (1 - \text{Prior}) \times (1 - \text{Likelihood})}\]

10.2.2 The Professor’s Soccer Analogy

NoteBayesian Updating in Sport

Professor’s note: “Two player teams are playing a knockout match… the home team in the league usually wins 60% of the matches. So in Bayesian statistics, you call this a prior belief…”

  • Starting belief (prior): Home team wins 60% of matches.
  • New evidence: Team A has a star striker who scores a lot — when he plays, the team wins 70%.
  • Updated belief (posterior): 0.78 (78% chance of winning).
  • More evidence: Key defender is injured — alone this suggests only 45% chance of winning.
  • Final updated belief: 0.74 (74% chance of winning).

Bayesian thinking = combining independent pieces of evidence sequentially.

10.2.3 Why “Naive”?

It assumes every feature is independent of every other feature — which is almost never true in practice. In the Titanic data, Fare and Pclass are obviously correlated, but Naive Bayes pretends they aren’t.

library(e1071)
nb_model <- naiveBayes(Survived ~ ., data = train_data)
pr <- predict(nb_model, newdata = test_data, type = "raw")

10.3 Support Vector Machines (SVM)

10.3.1 The Big Idea

Find the best-separating boundary between two classes — specifically, the one that leaves the widest possible gap (margin) between the closest points of each class.

10.3.2 Two Key Concepts

  1. Maximum margin — among all possible separating lines, pick the one furthest from any point. The points closest to the boundary are called support vectors.
  2. The kernel trick — map the data into a higher-dimensional space so that patterns that aren’t linearly separable in 2D become separable in higher dimensions. Conceptually similar to polynomial regression.
NoteWhy SVM Is Robust to Outliers

Professor’s note: “It really only is going to focus on [the support vectors at the boundary]. They really are the rows that are going to determine how it’s going to separate the data.”

Points far from the boundary don’t influence the model at all. So one or two wild outliers have no effect — unlike logistic regression, where every point contributes.

library(kernlab)
svm_model <- ksvm(Survived ~ ., data = train_data,
                  kernel = "rbfdot", prob.model = TRUE)

10.4 Neural Networks

10.4.1 The Big Idea

A network of nodes arranged in layers. Each node does a small mathematical calculation (weighted sum + nonlinear activation function) and passes its result to the next layer. By stacking many layers, the network can approximate almost any function.

10.4.2 How Training Works

Step What happens
1. Forward pass Inputs flow through the layers, producing a prediction.
2. Calculate error Compare prediction to actual value using a cost function.
3. Backpropagation Adjust every weight slightly, working backwards, to reduce the error.
4. Repeat Loop forward + backward hundreds or thousands of times.

10.4.3 Practical Constraints

  • Requires numeric input — categorical variables must be one-hot encoded.
  • Requires scaling — all numeric features should be mapped to a 0–1 range.
  • Slow and unstable on small datasets.
Warning⚠️ The Professor’s Warning

Professor’s note: “In practice, what I basically say about neural networks is: don’t use neural networks typically… I hardly ever use them outside of this class. I don’t think I’ve ever touched neural networks on tabular data unless you have a large amount of data, or it is something that is designed like a computer vision problem or [NLP] problem.”

For tabular data, Random Forest and SVM almost always outperform neural networks unless you have millions of rows. Save neural networks for image/text data.

10.5 Random Forest (Ensemble Method)

10.5.1 The Big Idea

Rather than building one decision tree, build hundreds of trees — each trained on slightly different data and features — and take the majority vote.

10.5.2 The Statistical Foundation — Central Limit Theorem

NoteAnalogy — Guessing Marbles in a Jar

Professor’s note: “If I ask everyone here in this class how many marbles are in the jar, you could be off by 20 or 30%. If I asked 100 people their guess on how many marbles are in the jar, and I took the average of that, it would be within 1 or 2% of the actual value.”

The average of many imperfect guesses is almost always better than any individual guess — the Central Limit Theorem. Random Forest exploits this by averaging many mediocre decision trees.

10.5.3 How Random Forest Creates Diverse Trees

If every tree saw the same data and features, they’d all be identical and the vote would be pointless. Random Forest uses three tricks to force diversity:

Trick What it does Why
Bagging (Bootstrap Aggregating) Each tree gets a random sample of rows, with replacement (~63% unique) Different rows = different trees
Random feature selection At each split, only a random subset of features is considered Stops one dominant feature (like gender) being chosen every time
Shallow trees Each tree is kept small (depth 1–2) Prevents any single tree from overfitting
NoteWhy Intentionally Weaken the Base Models?

Professor’s note: “If a few base models are very poor, then they will just be outvoted by the other base models.”

Each individual tree is kind of useless — but together, they make a very strong ensemble. Think wisdom of crowds.

10.5.4 Random Forest Feature Importance (Titanic)

The Random Forest also tells you which features matter most:

Sex > Fare > Age > Pclass > SibSp > Parch > Embarked

Warning⚠️ Different Algorithms, Different Opinions

Professor’s note: “For Random Forest, it’s gender, fair, age are the three most important variables, just going to show that different algorithms have different ways of deciding which are the most important features within your data set. When you have highly correlated variables like fair and ticket class, one algorithm might prioritise one of those columns over the others.”

Logistic regression put Pclass ahead of Fare; Random Forest does the opposite. Feature importance is algorithm-specific — don’t treat it as absolute truth.

library(randomForest)

11 Part 11 — Sampling: With vs Without Replacement

An important supporting concept for understanding bagging.

11.1 Without Replacement (The Default)

Like drawing bingo balls: once a number is called, it’s set aside and can’t come up again. All selected items are unique.

# Select 90 values from 1..100 WITHOUT replacement
tmp <- sample(100, 90)
length(unique(tmp))   # 90 — all unique

11.2 With Replacement

Like drawing a marble, noting its number, and putting it back before the next draw. Items can repeat.

# Select 90 values from 1..90 WITH replacement
tmp <- sample(90, 90, replace = TRUE)
length(unique(tmp))   # approximately 57 unique (~63%)
Important⭐ Key Number to Remember — The 63% Rule

When sampling n items from n with replacement, approximately 63% will be unique and ~37% will be duplicates. This is the mathematical basis of bagging in Random Forest.

The ~37% that get left out are called out-of-bag (OOB) samples and can be used to estimate error without needing a separate validation set.

# Part 1 — Why Accuracy Alone Is Not Enough

## The Problem With Accuracy

Accuracy is the most obvious measure: the percentage of predictions that were correct.

$$\text{Accuracy} = \frac{\text{Correct Predictions}}{\text{Total Predictions}} = \frac{TP + TN}{TP + TN + FP + FN}$$

It is easy to understand — 100% means every prediction was right. But there are **two situations where accuracy can fool you badly**.

### Situation 1 — Unequal Cost of Errors

In many real-world problems, the two types of errors are not equally bad.

::: {.callout-note icon=true}
## Example — Medical Diagnosis

- A **false positive** (predicting "sick" when the person is healthy) is **upsetting but not life-threatening** — the patient might get further tests and eventually be cleared.
- A **false negative** (predicting "healthy" when the person is actually sick) can be **fatal**, because treatment is delayed.

These two errors are *not* the same, but accuracy treats them as identical.
:::

::: {.callout-important icon=true}
## ⭐ Professor's real-world example — The Irish BreastCheck scandal

*Professor's note:* **"We had a scandal in the medical industry for our BreastCheck, which was an early screening programme to detect cancers... it probably managed to diagnose correctly whether a person was sick or healthy 95, 98, 99%. But it missed a number of people who had cancerous tumours and subsequently missed out on treatment, and by the time their cases progressed and they got the treatment, their cancers had spread, and the number of people unfortunately died from it."**

The professor's point: *"It doesn't really matter... whether you're 99% accurate or not. It's the number of people that are diagnosed as healthy but in fact they're sick. That's the only metric that you're worried about."*

**This kind of story is exactly the type of thing the professor uses to build an exam question around — be ready to explain why recall matters more than accuracy in medical screening.**
:::

### Situation 2 — Imbalanced Datasets

When one class is much more common than the other, a lazy model can score very high on accuracy while being useless.

::: {.callout-warning icon=true}
## Common Mistake

If 1% of transactions are fraudulent, a model that predicts "not fraud" for every transaction is **99% accurate** — and **catches zero fraud cases**. Accuracy looks great; the model is worthless.

*Professor's note:* **"A machine learning model could just turn on and basically say, 100% everyone's healthy, and I wouldn't catch those records. So in a lot of cases, accuracy is actually a misleading evaluation metric to use."**
:::

Real examples of imbalanced data the professor mentioned:

| Domain | Positive class (what we want to catch) | Typical prevalence |
|:---|:---|:---|
| Credit card fraud | Fraudulent transactions | < 0.1% |
| Medical screening (cancer) | Person has disease | ~5% |
| Spam filtering | Spam emails | Varies, often a minority |
| Manufacturing defects | Defective items | < 1% |

# Part 2 — The Confusion Matrix

A **confusion matrix** is a 2×2 table that breaks predictions into four categories. It is the foundation of every other classification metric.

## The Four Outcomes

We always pick one class to call **Positive** (the thing we are trying to detect). The other is **Negative**.

|   | **Actual Positive (1)** | **Actual Negative (0)** |
|---|:---:|:---:|
| **Predicted Positive (1)** | True Positive (TP) ✅ | False Positive (FP) ❌ |
| **Predicted Negative (0)** | False Negative (FN) ❌ | True Negative (TN) ✅ |

::: {.callout-note icon=true}
## How to Remember "True" and "False"

- Words that **start with "True"** → the model was **correct**.
- Words that **start with "False"** → the model was **wrong**.
- The second word (Positive/Negative) tells you **what the model predicted**.

So "False Negative" = the model predicted Negative, but was wrong (i.e. the actual answer was Positive).
:::

## The Titanic Example (Used Throughout Class)

The decision tree model from Week 3 produced this confusion matrix (with `Survived = 1` as the Positive class, after re-ordering):

```
         actuals
pr         1    0
   1      17    4      ← Model predicted SURVIVED
   0      19   50      ← Model predicted DID NOT SURVIVE
```

Reading this:

| Cell | Count | Meaning |
|:---|:---:|:---|
| TP | 17 | Did survive **AND** the model said they survived ✅ |
| FP | 4 | Did **not** survive, but the model said they did ❌ |
| FN | 19 | Did survive, but the model said they didn't ❌ |
| TN | 50 | Did **not** survive **AND** the model agreed ✅ |

**Total accuracy** = (17 + 50) / 90 = 74.4%

::: {.callout-important icon=true}
## ⭐ Exam-Critical Detail — The R Default vs. The "Standard" Layout

R by default puts the class labelled `0` in the first row/column. Most textbooks and Python libraries put the **Positive class (1) first**, because that is the class we care most about.

*Professor's note:* **"Unfortunately R kind of reverses your confusion matrix... so we're going to reverse that, because we want to treat our first class, or class where it's equal to 1, as a primary class for calculating other metrics."**

The R code to flip the matrix is:
```r
actuals <- factor(test_data$Survived, levels=c(1,0))
pr      <- factor(pr, levels=c(1,0))
tab     <- table(pr, actuals)
```

If you are ever asked to calculate precision and recall from an R confusion matrix, **check which class is the positive one first**.
:::

# Part 3 — Precision, Recall, and F1

Three metrics to memorise. Each one answers a different question.

## Precision — "When the model says YES, how often is it right?"

$$\text{Precision} = \frac{TP}{TP + FP}$$

Precision looks at the **predicted** column. Of all the cases the model flagged as Positive, what fraction actually were Positive?

**Titanic example:** Precision = 17 / (17 + 4) = **0.81**

::: {.callout-note icon=true}
## Plain-English Analogy — Precision is like a fishing net's "keep rate"

If you cast a net and pull in 100 things, and 81 of them are fish (the rest are seaweed), your **precision is 81%**. The net is quite clean — when it catches something, it's usually a fish.
:::

## Recall (Sensitivity) — "Of all the real YES cases, how many did the model find?"

$$\text{Recall} = \frac{TP}{TP + FN}$$

Recall looks at the **actual** column. Of all the cases that truly were Positive, what fraction did the model catch?

**Titanic example:** Recall = 17 / (17 + 19) = **0.47**

::: {.callout-note icon=true}
## Plain-English Analogy — Recall is like a search party's "find rate"

If 100 people are lost in a forest and the search party brings back 47 of them, the **recall is 47%**. It doesn't matter how many false alarms there were — only how many of the real lost people they found.
:::

::: {.callout-important icon=true}
## ⭐ Mnemonic the Professor Gave for the Exam

*Professor's note:* **"Precision and recall are two things that a lot of people have a bit of confusion about, and it can be hard to remember the definitions. The way I try and remember recall is: recall just looks at the actual values, and precision looks at the predicted values."**

- **Recall** → look at the **row where the Actual = 1** (focus on actuals)
- **Precision** → look at the **row where the Predicted = 1** (focus on predictions)

Expect to calculate both by hand from a confusion matrix on the exam.
:::

## F1 Score — The Balanced Summary

$$F_1 = 2 \times \frac{\text{Precision} \times \text{Recall}}{\text{Precision} + \text{Recall}}$$

F1 is the **harmonic mean** of precision and recall. It gives a single number that only gets high when **both** precision and recall are high.

**Titanic example:** F1 = 2 × (0.81 × 0.47) / (0.81 + 0.47) ≈ **0.60**

::: {.callout-warning icon=true}
## Why Harmonic Mean and Not Simple Average?

Because the harmonic mean **punishes extreme imbalance**. Consider a model with Precision = 1.0 and Recall = 0.0:

- Simple average: 0.5 (looks okay)
- F1 score: **0.0** (correctly shows the model is broken)

*Professor's note:* **"If precision is 0.99 but the recall is 0.01, it just doesn't take an average, call it 50% and call it done. It basically will allocate a very low F1 score on that."**
:::

## Quick Reference Table

| Metric | Formula | Answers the question | When it matters most |
|:---|:---|:---|:---|
| **Accuracy** | (TP+TN)/Total | "Overall, how often right?" | Balanced data, equal costs |
| **Precision** | TP/(TP+FP) | "When I say YES, am I right?" | When false alarms are expensive |
| **Recall** | TP/(TP+FN) | "Did I catch all the real YES cases?" | When missing a case is expensive |
| **F1** | Harmonic mean of P & R | "Are both P and R good?" | Imbalanced data, both errors matter |

# Part 5 — ROC Curve and AUC

## What Is the ROC Curve?

The **Receiver Operating Characteristic (ROC)** curve is a plot that shows how well a classifier can distinguish between the two classes **across all possible thresholds**.

::: {.callout-note icon=true}
## Fun Historical Analogy

The ROC curve was invented for **radar operators in World War II** who had to decide whether a blip on the screen was a real enemy plane or just noise. That's why it's called "Receiver Operating Characteristic."
:::

- **Y-axis:** Sensitivity = Recall = TP / (TP + FN)
- **X-axis:** 1 − Specificity = FP / (FP + TN) = False Positive Rate

A perfect model would go straight up the left side and across the top — 100% recall with 0% false positives.

## AUC — Area Under the Curve

**AUC** is a single number summarising the ROC curve.

| AUC value | Interpretation |
|:---:|:---|
| 1.0 | Perfect classifier |
| 0.9 – 1.0 | Excellent |
| 0.8 – 0.9 | Good |
| 0.7 – 0.8 | Fair |
| 0.5 | No better than random guessing |
| < 0.5 | Worse than random (predictions inverted) |

**Titanic decision tree AUC:** 0.764 (decent but not great)

```r
library(tidymodels)
test_data$predicted <- pr[,2]
test_data$Survived <- factor(test_data$Survived, levels=c(1,0))
autoplot(roc_curve(test_data, Survived, predicted))
roc_auc(test_data, Survived, predicted)
```

# Part 6 — The Precision / Recall Trade-Off

## The 0.5 Threshold Is Not Sacred

Most classifiers output a **probability** between 0 and 1. By default we convert this to a class using:

> If `prob >= 0.5` → predict Positive  
> Else → predict Negative

But **0.5 is just a convention**. By moving the threshold, we can prioritise precision or recall.

::: {.callout-note icon=true}
## Threshold as a Volume Knob

Think of the threshold as a volume knob for how eager the model is to say "Positive":

- **Low threshold (e.g., 0.1)** — the model shouts "Positive!" at almost everything. Catches most real positives (**high recall**) but also raises lots of false alarms (**low precision**).
- **High threshold (e.g., 0.9)** — the model only says "Positive" when 90%+ sure. Rarely wrong when it does (**high precision**) but misses many real cases (**low recall**).
:::

## Actual Results from Lab 4a (Logistic Regression on Titanic)

| Threshold | Accuracy | Precision | Recall | Character |
|:---:|:---:|:---:|:---:|:---|
| 0.1 | ~65% | 0.54 | 0.92 | "Paranoid" — catches almost everyone, lots of false alarms |
| **0.5** | ~73% | 0.73 | 0.53 | Balanced default |
| 0.9 | ~64% | 1.00 | 0.11 | "Cautious" — only when very sure |

**You cannot push up both precision AND recall** — improving one almost always costs the other.

::: {.callout-important icon=true}
## ⭐ Exam Application

If the exam describes a scenario, you should be able to recommend a threshold:

| Scenario | Prioritise | Move threshold... |
|:---|:---:|:---:|
| Cancer screening — don't miss a case | Recall | **Lower** |
| Spam filter — don't send real email to junk | Precision | **Higher** |
| Fraud detection — catch as many as possible | Recall | **Lower** |
| Legal evidence — must be certain | Precision | **Higher** |
:::

# Part 7 — Logistic Regression

## What Is It?

**Logistic regression** is a **classification** algorithm (despite "regression" in the name). It takes the framework of linear regression but maps the output through a **sigmoid (logistic) function** so the result is a probability between 0 and 1.

Linear regression formula:
$$z = \beta_0 + \beta_1 x_1 + \beta_2 x_2 + \dots + \beta_n x_n$$

Sigmoid transformation:
$$P(y = 1) = \frac{1}{1 + e^{-z}}$$

The output is the probability that the observation belongs to the positive class.

## Why It Is Popular

- **Interpretable coefficients** — each $\beta$ tells you how the variable affects the probability.
- **Fast** to train, even on large data.
- **Statistical significance** is reported via p-values.
- A good **baseline** before trying fancy algorithms.

## Key Points From Class

### Interpreting the Coefficients

- **Negative coefficient** → higher values of that feature **decrease** the probability of survival.
- **Positive coefficient** → higher values **increase** the probability of survival.

*Professor's note:* **"If you look at the way the formula for logistic regression is ... if the coefficient is negative, that means that a higher value is an indication that the person did not survive. So in this case, Pclass is negative, which means the higher value for Pclass, the less likely somebody survived."**

### Handling Categorical Variables

When R sees a categorical variable like `Sex`:

- It creates **indicator variables** (`Sexmale` = 0 or 1).
- One category is **absorbed into the intercept** — typically the alphabetically first value.
- So `Sexfemale` has **no explicit coefficient** — it's wrapped into the intercept.

::: {.callout-note icon=true}
## Why the Intercept Is So Large

The intercept effectively represents the "baseline person" — in the Titanic case, a **female passenger who embarked at Cherbourg**. Both conditions are associated with higher survival, so the intercept comes out large and positive.
:::

### The Missing-Data Problem

::: {.callout-warning icon=true}
## ⚠️ Common Mistake — Don't Forget to Handle NA's!

Logistic regression **cannot handle missing values**. If you train a model using a column with NAs (like `Age` in Titanic), the `predict()` function will return **NA** for those rows — and you'll get a broken confusion matrix.

**Two options:**

1. **Drop the column** entirely (quick fix)
2. **Impute the missing values** (better: use the **mean of the training set only**, never the full dataset)

*Professor's note:* **"We only calculate the average of the age across the [training] split of the records... We don't take the average over all of them, because that would mean that we're leaking data into the [model]."**
:::

### Occam's Razor — Simpler Is Often Better

In the lab, removing `Embarked`, `Parch`, and `Fare` (all statistically insignificant) **improved** accuracy slightly — from 73% to 74%.

**Why?** These variables added noise without adding signal. Correlated features (like `Fare` and `Pclass`) don't both need to be in the model.

::: {.callout-important icon=true}
## ⭐ The Concept of Multicollinearity

A variable may appear "insignificant" in the summary **not because it has no relationship with the outcome, but because another variable in the model already captures the same information**.

Example: `Embarked` looks insignificant because `Pclass` already explains most of its predictive power (people from Cherbourg were more likely to be first class).
:::

# Part 8 — Evaluation Metrics for Regression

For numerical targets (predicting a price, a temperature, a spend) we use different metrics.

## The Four Regression Metrics

| Metric | Formula | Units | Key property |
|:---|:---|:---|:---|
| **MAE** | $\frac{1}{n}\sum \lvert y_i - \hat{y}_i \rvert$ | Same as y | Easy to explain, treats all errors equally |
| **MSE** | $\frac{1}{n}\sum (y_i - \hat{y}_i)^2$ | y² | Penalises big errors heavily |
| **RMSE** | $\sqrt{\text{MSE}}$ | Same as y | Penalises big errors, yet still interpretable |
| **R²** | Correlation(y, ŷ)² | Unitless 0–1 | Proportion of variance explained |

## MAE — Mean Absolute Error

$$\text{MAE} = \frac{1}{n} \sum_{i=1}^{n} |y_i - \hat{y}_i|$$

::: {.callout-note icon=true}
## Professor's Example — Customer Spending Model

*Professor's note:* **"If we said that we had a model that was trying to predict how much a person was going to spend in our store in the next month, and if it basically said that the mean absolute error was 15, then the model is accurate plus or minus 15 euros."**

MAE's big advantage: it is in the **same units** as the target. Easy to communicate to business.
:::

## MSE — Mean Squared Error

$$\text{MSE} = \frac{1}{n} \sum_{i=1}^{n} (y_i - \hat{y}_i)^2$$

Squaring has two effects:

1. Removes negatives (a -5 error and a +5 error both contribute 25).
2. **Big errors are punished more** than small ones (2² = 4 vs 5² = 25).

Good for models where you **really don't want huge misses**.

## RMSE — Root Mean Squared Error

$$\text{RMSE} = \sqrt{\text{MSE}}$$

Squaring puts the error back in the same units as `y`, but RMSE still penalises large errors more than MAE does. In practice, **RMSE ≥ MAE always**.

## R² — Coefficient of Determination

Tells you what **fraction of the variance** in the target is explained by the model.

- R² = 1 → perfect fit
- R² = 0 → model is no better than just predicting the mean
- R² < 0 → model is worse than just predicting the mean (yes, this can happen on test data!)

**For most regression tasks, R² and RMSE are the preferred metrics.**

# Part 9 — The Bias–Variance Trade-Off

This is **the single most important concept** in the week. The professor came back to it many times.

## The Two Evils

| Concept | What it means | Training error | Test error | Also called |
|:---|:---|:---:|:---:|:---|
| **Bias** | Model too simple; misses the pattern | High | High | Underfitting |
| **Variance** | Model too complex; memorises noise | Low | **High** | Overfitting |

::: {.callout-note icon=true}
## Analogy — The Student Studying for an Exam

- **High bias (underfit)** — the student only skimmed the textbook. They don't know the material. They do badly on the practice exam **and** the real exam.
- **Good fit** — the student studied properly and understands the concepts. They do well on both practice and real exams.
- **High variance (overfit)** — the student memorised every word of the practice exam answers. On the practice exam they score 100%. On the real exam (with slightly different questions) they score badly because they never learned the underlying concepts.
:::

## Visualising It (from Lab 4b)

Using synthetic data, three polynomial regression models were fitted:

| Model | Formula | R² (training) | MAE (training) | Real-world behaviour |
|:---:|:---|:---:|:---:|:---|
| 1 | $y = \beta_0 + \beta_1 x$ | 0.17 | 3.45 | **Underfits** — a straight line cannot capture the curve. |
| 2 | $y = \beta_0 + \beta_1 x + \beta_2 x^2 + \beta_3 x^3$ | 0.79 | 1.54 | **Good fit** — captures the general shape. |
| 3 | Degree 6 polynomial | 0.97 | 0.62 | **Overfits** — hits every training point but oscillates wildly outside the training range. |

::: {.callout-warning icon=true}
## ⚠️ The Deceptive "Best" Model

Model 3 has the **best training metrics** (R² = 0.97, lowest MAE). It looks like the winner. **But it would fail catastrophically on new data**, because it has memorised the random noise in the training set, not the underlying relationship.

**This is the most common failure mode in applied machine learning.** Never trust training metrics alone.
:::

## How to Detect Overfitting

Three practical tests:

1. **Compare training vs test performance.** A big gap (great on training, mediocre on test) means the model overfit.
2. **Change the random seed** and refit. If results vary wildly, the model is unstable.
3. **Use cross-validation** (covered in a later class) for a systematic check.

# Part 10 — Survey of Other ML Algorithms

The professor gave a high-level tour of four more algorithms. Each uses a different "philosophy" to make predictions. All were tested on the Titanic data.

## Summary Comparison Table

| Algorithm | Core idea | Titanic accuracy | Strengths | Weaknesses |
|:---|:---|:---:|:---|:---|
| **Naive Bayes** | Update beliefs with evidence using Bayes' theorem | ~72% | Very fast, scalable, great baseline | Assumes features are independent (rarely true) |
| **SVM** | Draw the widest possible gap between classes | ~76% | Robust to outliers, good default | Slow on large datasets |
| **Neural Network** | Network of nodes learning non-linear patterns | ~75–78% | Approximates any function | Needs huge data, hard to interpret |
| **Random Forest** | Average the votes of 100+ decision trees | **~78%** | Best out-of-the-box on tabular data | Less interpretable than a single tree |

## Naive Bayes

### The Big Idea

Start with a **prior belief**, then **update** it each time new evidence arrives.

$$\text{Posterior} = \frac{\text{Prior} \times \text{Likelihood}}{\text{Prior} \times \text{Likelihood} + (1 - \text{Prior}) \times (1 - \text{Likelihood})}$$

### The Professor's Soccer Analogy

::: {.callout-note icon=true}
## Bayesian Updating in Sport

*Professor's note:* **"Two player teams are playing a knockout match... the home team in the league usually wins 60% of the matches. So in Bayesian statistics, you call this a prior belief..."**

- **Starting belief (prior):** Home team wins 60% of matches.
- **New evidence:** Team A has a star striker who scores a lot — when he plays, the team wins 70%.
- **Updated belief (posterior):** 0.78 (78% chance of winning).
- **More evidence:** Key defender is injured — alone this suggests only 45% chance of winning.
- **Final updated belief:** 0.74 (74% chance of winning).

Bayesian thinking = combining independent pieces of evidence sequentially.
:::

### Why "Naive"?

It assumes every feature is **independent** of every other feature — which is almost never true in practice. In the Titanic data, `Fare` and `Pclass` are obviously correlated, but Naive Bayes pretends they aren't.

```r
library(e1071)
```

## Support Vector Machines (SVM)

### The Big Idea

Find the **best-separating boundary** between two classes — specifically, the one that leaves the **widest possible gap** (margin) between the closest points of each class.

### Two Key Concepts

1. **Maximum margin** — among all possible separating lines, pick the one furthest from any point. The points closest to the boundary are called **support vectors**.
2. **The kernel trick** — map the data into a **higher-dimensional space** so that patterns that aren't linearly separable in 2D become separable in higher dimensions. Conceptually similar to polynomial regression.

::: {.callout-note icon=true}
## Why SVM Is Robust to Outliers

*Professor's note:* **"It really only is going to focus on [the support vectors at the boundary]. They really are the rows that are going to determine how it's going to separate the data."**

Points far from the boundary don't influence the model at all. So one or two wild outliers have no effect — unlike logistic regression, where every point contributes.
:::

```r
library(kernlab)

                  kernel = "rbfdot", prob.model = TRUE)
```

## Neural Networks

### The Big Idea

A **network of nodes** arranged in **layers**. Each node does a small mathematical calculation (weighted sum + nonlinear activation function) and passes its result to the next layer. By stacking many layers, the network can approximate almost any function.

### How Training Works

| Step | What happens |
|:---:|:---|
| 1. Forward pass | Inputs flow through the layers, producing a prediction. |
| 2. Calculate error | Compare prediction to actual value using a **cost function**. |
| 3. Backpropagation | Adjust every weight slightly, working backwards, to reduce the error. |
| 4. Repeat | Loop forward + backward hundreds or thousands of times. |

### Practical Constraints

- **Requires numeric input** — categorical variables must be **one-hot encoded**.
- **Requires scaling** — all numeric features should be mapped to a 0–1 range.
- **Slow and unstable** on small datasets.

::: {.callout-warning icon=true}
## ⚠️ The Professor's Warning

*Professor's note:* **"In practice, what I basically say about neural networks is: don't use neural networks typically... I hardly ever use them outside of this class. I don't think I've ever touched neural networks on tabular data unless you have a large amount of data, or it is something that is designed like a computer vision problem or [NLP] problem."**

For tabular data, Random Forest and SVM almost always outperform neural networks unless you have millions of rows. Save neural networks for image/text data.
:::

## Random Forest (Ensemble Method)

### The Big Idea

Rather than building one decision tree, build **hundreds of trees** — each trained on slightly different data and features — and **take the majority vote**.

### The Statistical Foundation — Central Limit Theorem

::: {.callout-note icon=true}
## Analogy — Guessing Marbles in a Jar

*Professor's note:* **"If I ask everyone here in this class how many marbles are in the jar, you could be off by 20 or 30%. If I asked 100 people their guess on how many marbles are in the jar, and I took the average of that, it would be within 1 or 2% of the actual value."**

The average of many imperfect guesses is almost always better than any individual guess — the **Central Limit Theorem**. Random Forest exploits this by averaging many mediocre decision trees.
:::

### How Random Forest Creates Diverse Trees

If every tree saw the same data and features, they'd all be identical and the vote would be pointless. Random Forest uses **three tricks** to force diversity:

| Trick | What it does | Why |
|:---|:---|:---|
| **Bagging** (Bootstrap Aggregating) | Each tree gets a **random sample of rows, with replacement** (~63% unique) | Different rows = different trees |
| **Random feature selection** | At each split, only a **random subset of features** is considered | Stops one dominant feature (like gender) being chosen every time |
| **Shallow trees** | Each tree is kept small (depth 1–2) | Prevents any single tree from overfitting |

::: {.callout-note icon=true}
## Why Intentionally Weaken the Base Models?

*Professor's note:* **"If a few base models are very poor, then they will just be outvoted by the other base models."**

Each individual tree is kind of useless — but together, they make a very strong ensemble. Think wisdom of crowds.
:::

### Random Forest Feature Importance (Titanic)

The Random Forest also tells you which features matter most:

**Sex > Fare > Age > Pclass > SibSp > Parch > Embarked**

::: {.callout-warning icon=true}
## ⚠️ Different Algorithms, Different Opinions

*Professor's note:* **"For Random Forest, it's gender, fair, age are the three most important variables, just going to show that different algorithms have different ways of deciding which are the most important features within your data set. When you have highly correlated variables like fair and ticket class, one algorithm might prioritise one of those columns over the others."**

Logistic regression put `Pclass` ahead of `Fare`; Random Forest does the opposite. **Feature importance is algorithm-specific — don't treat it as absolute truth.**
:::

# Part 11 — Sampling: With vs Without Replacement

An important supporting concept for understanding bagging.

## Without Replacement (The Default)

Like drawing bingo balls: once a number is called, it's set aside and can't come up again. All selected items are unique.

```r
# Select 90 values from 1..100 WITHOUT replacement
tmp <- sample(100, 90)
length(unique(tmp))   # 90 — all unique
```

## With Replacement

Like drawing a marble, noting its number, and putting it **back** before the next draw. Items can repeat.

```r
# Select 90 values from 1..90 WITH replacement
tmp <- sample(90, 90, replace = TRUE)
length(unique(tmp))   # approximately 57 unique (~63%)
```

::: {.callout-important icon=true}
## ⭐ Key Number to Remember — The 63% Rule

When sampling **n items from n with replacement**, approximately **63% will be unique** and ~37% will be duplicates. This is the mathematical basis of bagging in Random Forest.

The ~37% that get left out are called **out-of-bag (OOB)** samples and can be used to estimate error without needing a separate validation set.
:::
